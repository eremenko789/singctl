# Feature Specification: API Adapter & Auth

**Feature Branch**: `004-api-adapter-auth`

**Created**: 2026-07-16

**Status**: Draft

**Input**: User description: "F04 API adapter & auth — тонкий internal/api/ над codegen: Bearer-авторизация, base URL, timeout, маппинг ответов. Depends F02, F03. Out of scope: команды CLI; retry (F05). Acceptance: адаптер принимает токен/URL из конфига; unit-тесты с мок-HTTP (минимум 1 happy path)."

## Clarifications

### Session 2026-07-16

- Q: Какова поверхность адаптера в F04? → A: Фабрика сеанса (Bearer + base URL + timeout) + общий хелпер маппинга успеха/неуспеха (как минимум HTTP status); happy-path тест вызывает одну codegen-операцию через сеанс; полноценные фасады всех сущностей — вне F04 (CLI/TUI позже).
- Q: Входит ли удалённая проверка `config validate` в DoD F04? → A: MUST: при наличии токена `config validate` выполняет один лёгкий вызов API через адаптер; при сбое — не «удалённо OK»; без CRUD-сущностей.
- Q: Когда отклонять пустой токен? → A: Fail-fast: фабрика сеанса сразу отклоняет пустой/whitespace токен (сеанс без credentials не создаётся).
- Q: Каков контракт ошибки не-2xx для передачи в F05? → A: Типизированная ошибка с доступным HTTP status (+ опционально body/сообщение); полная taxonomy кодов — F05 (через errors.As / аналог), без парсинга строк.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Build authenticated API session from config (Priority: P1)

Разработчик (и последующие CLI/TUI-команды) получает готовый к запросам API-сеанс из действующей конфигурации: base URL, токен и timeout применяются автоматически, без ручной сборки HTTP-клиента в каждой команде.

**Why this priority**: Без единого адаптера поверх codegen нельзя безопасно вызывать API; это фундамент для `config validate`, CRUD-команд и TUI (constitution III–IV).

**Independent Test**: Передать в фабрику адаптера известные значения base URL, токена и timeout; убедиться, что созданный сеанс готов к вызовам и использует именно эти параметры (проверка через мок-HTTP или инспекцию исходящего запроса).

**Acceptance Scenarios**:

1. **Given** effective-конфиг с `api.base_url`, «голым» `api.token` и `api.timeout`, **When** создаётся API-адаптер из этой конфигурации, **Then** адаптер настроен на указанный base URL и timeout и готов выполнять аутентифицированные запросы.
2. **Given** токен задан runtime-override (как глобальный `--token`), а в файле другой токен, **When** создаётся адаптер из effective settings, **Then** в исходящих запросах используется override-токен, не значение из файла.
3. **Given** токен отсутствует или состоит только из пробелов, **When** вызывается фабрика сеанса, **Then** фабрика возвращает понятную ошибку и сеанс не создаётся (без сетевого запроса), сообщение пригодно для подсказки `config set-token`.

---

### User Story 1b - Remote config validate via adapter (Priority: P1)

Пользователь выполняет `singctl config validate` и получает реальную проверку доступности API через адаптер (не локальную заглушку), без выполнения CRUD сущностей.

**Why this priority**: DoD F04 (clarification): validate — первый end-to-end потребитель сеанса; иначе адаптер доказан только unit-моками.

**Independent Test**: С мок HTTP (или тестовым сервером) и токеном в конфиге выполнить `config validate` — успех при 2xx; при 401/сетевой ошибке — ненулевой код и честное сообщение без ложного «удалённо OK».

**Acceptance Scenarios**:

1. **Given** токен задан и мок/API отвечает успехом на лёгкий вызов адаптера, **When** пользователь выполняет `singctl config validate`, **Then** команда завершается кодом 0 и сообщает об успешной удалённой проверке (не текст заглушки).
2. **Given** токен задан, но API возвращает неуспех или недоступен, **When** выполняется `config validate`, **Then** ненулевой код и сообщение об ошибке; MUST NOT утверждать успешную удалённую проверку.
3. **Given** токен отсутствует, **When** выполняется `config validate`, **Then** ненулевой код и подсказка `config set-token` (как в F02), без сетевого запроса — в том числе потому что фабрика сеанса fail-fast не создаёт сеанс.

### User Story 2 - Authenticated happy-path request (Priority: P1)

Потребитель адаптера (тест или будущая команда) выполняет одну типичную операцию API через адаптер и получает расшифрованный успешный ответ — с корректным Bearer-заголовком и без ручного HTTP.

**Why this priority**: Acceptance F04 требует минимум один happy path с мок-HTTP; это доказывает, что auth + base URL + маппинг успеха работают вместе.

**Independent Test**: Поднять мок HTTP-сервер, отвечающий успешным телом на одну известную операцию; создать адаптер с base URL мока и тестовым токеном; выполнить вызов через адаптер и сверить результат с ожидаемой моделью ответа; проверить, что ушёл заголовок `Authorization: Bearer <token>`.

**Acceptance Scenarios**:

1. **Given** мок API отвечает успехом (2xx) с валидным телом на выбранную операцию, **When** адаптер выполняет этот вызов с тестовым токеном, **Then** вызывающий код получает успешно смаппленный результат (типизированные данные операции), а в запросе присутствует `Authorization: Bearer <голый-токен>`.
2. **Given** в конфиге хранится токен без префикса `Bearer`, **When** адаптер формирует запрос, **Then** заголовок содержит ровно один префикс `Bearer` (нет `Bearer Bearer …`).
3. **Given** мок не отвечает дольше заданного timeout, **When** выполняется вызов через адаптер, **Then** вызов завершается ошибкой таймаута/контекста (без зависания), пригодной для последующей обработки.

---

### User Story 3 - Surface non-success responses without retry (Priority: P2)

Потребитель адаптера при неуспешном HTTP-ответе получает структурированную ошибку с доступным статусом (и при возможности телом/сообщением), без автоматических повторов — чтобы F05 мог нарастить taxonomy и backoff поверх этого контракта.

**Why this priority**: «Маппинг ответов» включает не только успех; но детальная классификация 401/403/404/422/429/5xx и retry — F05. В F04 достаточно честно прокинуть неуспех.

**Independent Test**: Мок возвращает 4xx/5xx; вызов/маппинг завершается типизированной ошибкой, из которой программно читается HTTP-статус; повторных запросов к моку нет.

**Acceptance Scenarios**:

1. **Given** мок возвращает не-2xx (например, 401 или 404), **When** адаптер выполняет операцию / маппинг ответа, **Then** вызывающий код получает типизированную ошибку неуспеха, из которой программно читается HTTP status (не только текст), а не «успешный пустой» результат.
2. **Given** мок возвращает 429 или 5xx, **When** адаптер выполняет операцию в scope F04, **Then** выполняется ровно одна попытка запроса (без retry/backoff) и возвращается та же типизированная ошибка со status.

---

### User Story 4 - Shared adapter boundary for CLI and TUI (Priority: P2)

Будущие CLI- и TUI-слои получают API-сеанс из одного адаптерного слоя (фабрика + общий маппинг ответов), а не собирают auth/base URL/timeout в каждом UI-пакете. Полные фасады сущностей появятся вместе с командами; в F04 граница — сеанс и хелперы, не обёртки всех operations.

**Why this priority**: Constitution IV; ценность проявляется при следующих фичах, но граница пакета должна быть заложена сейчас.

**Independent Test**: Убедиться по структуре поставки, что фабрика сеанса и маппинг живут в `internal/api/`, а не размазаны по `internal/cli/`; unit-тесты адаптера не требуют cobra/TUI и не требуют фасадов всех 51 операций.

**Acceptance Scenarios**:

1. **Given** собранный адаптерный пакет, **When** ревьюер смотрит зависимости тестов happy path, **Then** тесты мокают HTTP на уровне адаптера и не поднимают CLI/TUI.
2. **Given** сгенерированный клиент в `internal/apiclient/`, **When** выполняется аутентифицированный вызов через сеанс адаптера, **Then** транспорт/вызов идёт через codegen-клиент (типы/методы codegen), без нового ручного CRUD/DTO и без полного фасада всех operations в F04.

---

### Edge Cases

- Пустой или whitespace-only токен: фабрика сеанса fail-fast — ошибка конфигурации, сеанс не создаётся, сеть не вызывается.
- Пустой `base_url`: отказ при создании/вызове с понятной ошибкой (не «успех» на неверном хосте).
- Невалидный `timeout` в конфиге (уже отклонённый F02 при `set`) — если effective значение битое: понятная ошибка при создании адаптера.
- Токен в тестовых фикстурах — только явно фиктивный низкоэнтропийный (`test-token-…` / `fake-…`); реальные секреты и обход gitleaks запрещены (constitution VII).
- Ответ 2xx с телом, не соответствующим ожидаемой схеме операции: ошибка декодирования/маппинга, не «тихий» успех с нулевыми данными без сигнала.
- Сетевая ошибка (connection refused к моку/хосту): ошибка транспорта без retry в F04.
- `Authorization` и токен MUST NOT попадать в обычные user-facing логи адаптера; debug-redact углублённо — F29.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST предоставлять тонкий API-адаптерный слой поверх сгенерированного клиента (`internal/api/` над `internal/apiclient/`): фабрика сеанса + общий маппинг ответов; без ручных HTTP CRUD/DTO сущностей и без полного фасада всех operations в F04.
- **FR-001a**: Адаптер MUST предоставлять фабрику сеанса, которая конфигурирует codegen-клиент (base URL, Bearer, timeout) и отдаёт потребителю доступ к codegen-операциям через этот сеанс.
- **FR-001b**: Адаптер MUST предоставлять общий хелпер(ы) маппинга: успешный 2xx → типизированный результат; не-2xx → типизированная ошибка с программно доступным HTTP status и опционально телом/сообщением (детальная taxonomy кодов — F05).
- **FR-002**: Адаптер MUST принимать параметры сеанса из effective-конфигурации (F02): как минимум `base_url`, `token`, `timeout`.
- **FR-003**: При формировании заголовка `Authorization` адаптер MUST добавлять префикс `Bearer` (с пробелом) к «голому» токену из конфига/override; MUST NOT дублировать префикс, если значение уже начинается с `Bearer `.
- **FR-004**: Адаптер MUST применять заданный timeout ко исходящим запросам сеанса (через HTTP-клиент/контекст — на уровне plan).
- **FR-005**: Happy-path / DoD F04 MUST демонстрировать как минимум одну репрезентативную codegen-операцию, выполненную через сеанс адаптера, с маппингом успешного (2xx) ответа; MUST NOT требовать обёрток всех 51 операций.
- **FR-006**: При не-2xx ответе общий маппинг MUST возвращать типизированную ошибку с программно доступным HTTP status (поле или метод; пригодно для `errors.As` / аналога в F05); MUST NOT полагаться только на парсинг текста сообщения; MUST NOT выполнять automatic retry/backoff (это F05).
- **FR-006a**: Типизированная ошибка не-2xx MAY включать тело ответа или краткое сообщение API, если оно доступно без отдельного ручного DTO-слоя сущностей.
- **FR-007**: Фабрика сеанса MUST отклонять пустой или whitespace-only токен сразу (fail-fast): сеанс не создаётся, сетевой запрос не выполняется; ошибка понятна и пригодна для подсказки `config set-token`.
- **FR-007a**: Система MUST NOT создавать «сеанс без credentials» для обычного пути `NewFromConfig` / эквивалента в F04 (lazy-отказ при первом вызове — вне выбранного контракта).
- **FR-008**: Система MUST включать unit-тесты адаптера с мок-HTTP: минимум один happy path (успешный ответ + проверка Bearer) и минимум один неуспешный ответ без retry.
- **FR-009**: Система MUST NOT добавлять новые пользовательские CLI-команды сущностей, MUST NOT реализовывать retry/exponential backoff и MUST NOT добавлять полный фасад всех API-операций в рамках F04.
- **FR-010**: Адаптерный слой MUST быть канонической фабрикой сеанса (auth/base URL/timeout + маппинг) для последующих CLI и TUI (constitution IV); UI-пакеты не должны заново собирать эти параметры в обход адаптера.
- **FR-011**: Тестовые фикстуры токенов MUST соответствовать constitution VII (фиктивные, низкоэнтропийные, без gitleaks allowlist).
- **FR-012**: В DoD F04 `config validate` MUST при наличии токена выполнять удалённую проверку через адаптер (один лёгкий вызов, без CRUD сущностей); при успехе — код 0 и явное сообщение об удалённой проверке; при сбое сети/не-2xx — ненулевой код без ложного «удалённо OK»; при отсутствии токена — поведение F02 (ошибка + подсказка `set-token`, без сети).
- **FR-012a**: Удалённая проверка validate MUST использовать тот же сеанс/маппинг, что и остальной адаптер (не отдельный ручной HTTP-клиент).

### Key Entities

- **API Session / Adapter**: Фабрика и сеанс доступа к SingularityApp REST: base URL, credentials, timeout, доступ к codegen-операциям и общий маппинг ответов; не полный фасад всех entities в F04.
- **Effective API Settings**: Снимок `base_url` / `token` / `timeout` после резолвинга конфига и runtime-override (из F02).
- **Bearer Credential**: Значение для `Authorization`, полученное из «голого» токена добавлением префикса `Bearer`.
- **Mapped Success Result**: Типизированный результат успешной операции (модели codegen), возвращаемый потребителю адаптера.
- **HTTP Failure Signal**: Типизированная ошибка неуспешного HTTP-ответа с программно доступным status (+ опционально body/сообщение); без taxonomy по кодам и без retry в F04. Транспортные ошибки (timeout, connection refused) — отдельные ошибки без HTTP status.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: В 100% прогонов unit-теста happy path адаптер завершает вызов успешно при 2xx от мока и передаёт в запросе заголовок `Authorization: Bearer <test-token>`.
- **SC-002**: Адаптер, созданный из конфига с заданными base URL и timeout, в 100% проверяемых исходящих запросах обращается к этому base URL и соблюдает timeout (нет «вечного» зависания в тесте с укороченным timeout).
- **SC-003**: При мок-ответе 4xx/5xx адаптер в 100% случаев возвращает типизированную ошибку, из которой тест программно читает HTTP status, и выполняет ровно одну HTTP-попытку (0 retry).
- **SC-004**: При пустом/whitespace токене фабрика сеанса в 100% случаев возвращает ошибку и не создаёт сеанс; секреты в сообщениях об ошибке не утекают.
- **SC-005**: В поставке F04 нет новых CLI-команд бизнес-CRUD, нет полного фасада всех operations и нет логики retry/backoff — 0 таких добавлений в scope.
- **SC-006**: Разработчик может прогнать unit-тесты адаптера с мок-HTTP офлайн (без реального SingularityApp) одной канонической командой тестов проекта (`make test`) и получить зелёный happy path.
- **SC-007**: `config validate` с токеном и успешным мок/ответом API в 100% автоматизированных прогонов завершается кодом 0 без текста заглушки; при смоделированном сбое API — в 100% случаев ненулевой код и отсутствие утверждения «удалённо OK».

## Assumptions

- Фича зависит от F02 (резолвинг конфига, «голый» токен, effective settings, `--token`) и F03 (сгенерированный клиент в `internal/apiclient/`).
- Канонический каталог адаптера — `internal/api/` (как в backlog F04 и constitution III «тонкий адаптер»).
- Поверхность F04: фабрика сеанса + общий маппинг; не обёртки всех operations (clarification 2026-07-16).
- Репрезентативная операция для happy path / validate — `ProjectController_list` (`GET /v2/project`) через `ProjectControllerListWithResponse` (зафиксировано в plan/research F04).
- «Маппинг ответов» в F04 = общий хелпер(ы): декодирование успеха в типы codegen + типизированная ошибка не-2xx со status (+ опц. body); детальная taxonomy ошибок и UX-тексты по кодам — F05 поверх этого контракта.
- Default base URL / timeout совпадают с F02 (`https://api.singularity-app.ru`, `30s`), если не переопределены конфигом.
- Удалённая проверка `config validate` через адаптер — обязательная часть DoD F04 (clarification 2026-07-16); unit-тесты с мок-HTTP достаточны (реальный API не обязателен для CI).
- Пустой токен: fail-fast в фабрике сеанса (clarification 2026-07-16); lazy-отказ при вызове не используется.
- Контракт ошибки не-2xx: типизированный status для F05, не парсинг строк (clarification 2026-07-16).
- TUI и команды сущностей появятся позже и будут потреблять этот адаптер, не дублируя auth.
- Интеграционные тесты против реального API — вне обязательного DoD F04 (нужен токен из `.env`); достаточно мок-HTTP.

## Out of Scope

- Пользовательские CLI-команды сущностей (`task`, `project`, `habit`, …) и форматы вывода CRUD.
- Полный фасад/обёртки всех (или большинства) API-операций в адаптере — отложено до фич CLI/TUI поверх сеанса.
- Retry, exponential backoff, лимиты попыток для 429/5xx (F05).
- Детальный маппинг и пользовательские сообщения для 401/403/404/422/429/5xx как отдельная taxonomy (F05); в F04 — типизированная ошибка со status без классификации по кодам.
- TUI, debug-HTTP verbose redact (F29), secure-env политики (F30).
- Ручные HTTP-обёртки CRUD/DTO в обход codegen.
- Изменение пайплайна OpenAPI fetch/generate (F03), кроме потребления уже сгенерированного клиента.
- Обязательные integration-тесты с production/staging API (мок-HTTP для validate/адаптера достаточен).
- Оставление `config validate` как локальной заглушки после F04.
