# Feature Specification: OpenAPI Codegen Pipeline

**Feature Branch**: `003-openapi-codegen`

**Created**: 2026-07-16

**Status**: Draft

**Input**: User description: "F03 OpenAPI codegen pipeline — конфиг генерации API-клиента; Make-таргеты openapi-fetch, generate, api-coverage-check; клиент только из codegen в internal/apiclient/. Depends F01. Out of scope: ручные HTTP-обёртки; CLI-команды. Acceptance: make generate создаёт клиент; coverage-check совпадает с ожидаемым числом operations."

## Clarifications

### Session 2026-07-16

- Q: Обязателен ли коммит сгенерированного клиента в DoD F03? → A: MUST: в DoD F03 входит закоммиченный `internal/apiclient/*.gen.go` (+ конфиг и снимок).
- Q: Должен ли `make generate` жёстко зависеть от `api-coverage-check`? → A: Таргеты независимы; рекомендуемый порядок только в docs/`make help`.
- Q: Насколько строго `api-coverage-check` проверяет матрицу `coverage.md`? → A: Только ops в JSON == expected и файл `coverage.md` существует.
- Q: Нужна ли в F03 проверка «`make generate` не даёт git-diff»? → A: Не в F03: только успешный generate + коммит артефактов; drift-check — позже.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Generate typed API client from snapshot (Priority: P1)

Разработчик (или CI) по зафиксированному снимку OpenAPI получает готовый типизированный HTTP-клиент и модели сущностей API — без ручного написания CRUD/DTO — одной канонической командой генерации.

**Why this priority**: Constitution III запрещает ручные HTTP-обёртки; без работающей генерации невозможны F04+ (адаптер, CLI по сущностям). Это ядро пайплайна.

**Independent Test**: При наличии актуального снимка OpenAPI и конфига генерации выполнить `make generate` и убедиться, что в каталоге клиента появились сгенерированные артефакты, покрывающие операции снимка; повторный прогон идемпотентен по смыслу (успешный exit, клиент обновлён).

**Acceptance Scenarios**:

1. **Given** в репозитории есть снимок OpenAPI и конфиг генерации, **When** разработчик выполняет `make generate`, **Then** создаётся (или обновляется) сгенерированный клиент с моделями в каноническом каталоге `internal/apiclient/`, команда завершается успешно.
2. **Given** конфиг генерации отсутствует, **When** выполняется `make generate`, **Then** команда завершается с ненулевым кодом и понятным сообщением, что нужно создать конфиг (ссылка на docs допустима), без частичных «битых» артефактов как успешного результата.
3. **Given** успешная генерация уже выполнена, **When** снова выполняется `make generate` без изменения снимка, **Then** команда снова успешна; результат остаётся валидным клиентом (перегенерация допустима).

---

### User Story 2 - Refresh OpenAPI snapshot from upstream (Priority: P1)

Разработчик обновляет локальный снимок контракта API с официального upstream одной Make-командой, чтобы последующая генерация и проверка покрытия опирались на актуальный источник.

**Why this priority**: Снимок — вход codegen и матрицы покрытия; без автоматизированного fetch разработчики расходятся по устаревшим копиям (constitution VIII).

**Independent Test**: Выполнить `make openapi-fetch` (при доступности сети к upstream) и убедиться, что локальные JSON/YAML снимки обновлены и читаемы; затем `make api-coverage-check` отражает согласованность с ожидаемым числом операций.

**Acceptance Scenarios**:

1. **Given** заданы параметры окружения для base URL / URL снимка (через `.env` или значения по умолчанию из `.env.example`), **When** выполняется `make openapi-fetch`, **Then** локальные файлы снимка OpenAPI (JSON и YAML) перезаписываются содержимым upstream, команда завершается успешно.
2. **Given** upstream недоступен или возвращает ошибку, **When** выполняется `make openapi-fetch`, **Then** команда завершается с ненулевым кодом; репозиторий не считается успешно обновлённым (разработчик видит сбой fetch).

---

### User Story 3 - Verify API operation coverage count (Priority: P1)

Разработчик (или CI) проверяет, что снимок OpenAPI содержит ожидаемое число HTTP-операций и что матрица покрытия присутствует — без ручного подсчёта paths.

**Why this priority**: Acceptance F03 и constitution III требуют полного покрытия публичного REST; расхождение числа operations должно ловиться до merge.

**Independent Test**: На текущем снимке с известным ожиданием выполнить `make api-coverage-check` — успех при совпадении; при искусственно изменённом ожидании или повреждённом снимке — отказ.

**Acceptance Scenarios**:

1. **Given** снимок OpenAPI с числом operations, равным зафиксированному ожиданию (сейчас **51** для API v2), и присутствует матрица `docs/api/coverage.md`, **When** выполняется `make api-coverage-check`, **Then** команда завершается успешно и сообщает согласие с ожиданием.
2. **Given** число operations в снимке не равно ожидаемому, **When** выполняется `make api-coverage-check`, **Then** команда завершается с ненулевым кодом (CI/разработчик видят расхождение).
3. **Given** матрица покрытия отсутствует, **When** выполняется `make api-coverage-check`, **Then** команда завершается с ненулевым кодом.
4. **Given** `coverage.md` существует, но содержимое матрицы устарело относительно снимка (при верном числе ops в JSON), **When** выполняется `make api-coverage-check`, **Then** в scope F03 команда всё равно успешна — актуальность строк матрицы поддерживается процессом/ревью, не этим check.

---

### User Story 4 - Keep generated client and config in version control (Priority: P1)

Разработчик коммитит конфиг генерации, снимок OpenAPI, матрицу покрытия и сгенерированный клиент так, чтобы клон репозитория имел воспроизводимый клиент без обязательного немедленного обращения к upstream.

**Why this priority**: DoD F03 требует закоммиченный codegen-клиент: иначе офлайн-сборка/ревью диффа клиента невозможны; согласовано с `docs/openapi-codegen.md`.

**Independent Test**: После `make generate` убедиться, что артефакты из списка «что коммитить» присутствуют и не содержат секретов; `.env` не требуется для офлайн-перегенерации из уже лежащего снимка.

**Acceptance Scenarios**:

1. **Given** успешный `make generate` от закоммиченного снимка, **When** ревьюер смотрит поставку F03, **Then** в git присутствуют: конфиг генерации, снимок OpenAPI, матрица покрытия, сгенерированные файлы клиента; `.env` и реальные токены — нет.
2. **Given** свежий клон без сети к upstream, **When** выполняется `make generate` по уже имеющемуся снимку, **Then** клиент успешно пересобирается офлайн.

---

### Edge Cases

- Снимок OpenAPI повреждён / невалидный YAML или JSON: `generate` и/или `api-coverage-check` завершаются с ненулевым кодом и понятной ошибкой.
- Ожидаемое число operations устарело относительно upstream после `openapi-fetch`: `api-coverage-check` падает, пока ожидание и/или матрица покрытия не обновлены осознанно.
- Инструмент генерации не установлен в PATH: `make generate` падает с диагностируемой ошибкой (не «тихий» успех).
- Частичный снимок (только JSON или только YAML отсутствует после сбоя fetch): последующие проверки/генерация не должны считаться успешными на неполных данных.
- Повторная генерация не должна требовать ручных правок внутри `*.gen.go`; любые ручные HTTP CRUD/DTO в обход codegen — вне scope и запрещены constitution.
- Параметры URL/токена для fetch берутся из `.env` / defaults; отсутствие `.env` не должно ломать fetch, если defaults из Makefile/`.env.example` достаточны для публичного снимка.
- `make generate` при «красном» `api-coverage-check` (рассинхрон ожидания ops) остаётся допустимым: независимость таргетов позволяет чинить снимок/ожидание/клиент по отдельности.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST предоставлять Make-таргет `generate`, который по снимку OpenAPI и конфигу генерации создаёт типизированный API-клиент с моделями в `internal/apiclient/` (только codegen, без ручных HTTP-обёрток в этой фиче).
- **FR-002**: Система MUST включать конфиг генерации `api/oapi-codegen.yaml`, задающий пакет клиента, генерацию models+client и путь вывода в `internal/apiclient/` (как в `docs/openapi-codegen.md`).
- **FR-003**: Система MUST предоставлять Make-таргет `openapi-fetch`, обновляющий локальные снимки OpenAPI (JSON и YAML) из upstream API.
- **FR-004**: Система MUST предоставлять Make-таргет `api-coverage-check`, который (1) сверяет число HTTP-операций в снимке OpenAPI JSON с зафиксированным ожиданием и (2) проверяет **наличие** файла матрицы `docs/api/coverage.md`. В scope F03 check MUST NOT парсить строки матрицы и MUST NOT сверять operationId из снимка со строками `coverage.md`.
- **FR-005**: При совпадении числа operations с ожиданием и наличии `coverage.md` `api-coverage-check` MUST завершаться успешно; при расхождении числа или отсутствии матрицы — с ненулевым кодом.
- **FR-006**: Ожидаемое число operations для актуального снимка API v2 MUST быть **51** (как в `docs/api/coverage.md` / истории снимка), пока осознанно не обновлены снимок, матрица и ожидание вместе.
- **FR-007**: Сгенерированный клиент MUST покрывать все операции публичного REST из актуального снимка, используемого генерацией (пропуск endpoint без фиксации out-of-scope в Spec Kit запрещён на уровне constitution; в F03 проверяется через согласованность снимка и успешной генерации полного клиента).
- **FR-008**: Система MUST NOT добавлять ручные HTTP CRUD/DTO-обёртки и MUST NOT реализовывать CLI-команды сущностей в рамках этой фичи.
- **FR-009**: Повторяемые шаги fetch / generate / coverage-check MUST запускаться через Makefile как **независимые** таргеты (без Make-prerequisite `generate` → `api-coverage-check`); рекомендуемый порядок (`openapi-fetch` → `api-coverage-check` → `generate`) MUST быть зафиксирован в docs и/или `make help`; параметры окружения при необходимости читаются из `.env` (файл не коммитится); пример — `.env.example`.
- **FR-009a**: Система MUST NOT делать успешный `api-coverage-check` обязательным предусловием локального `make generate`.
- **FR-010**: В поставку F03 MUST входить закоммиченные: снимок OpenAPI, матрица покрытия, конфиг генерации и сгенерированный клиент (`internal/apiclient/*.gen.go`); MUST NOT коммититься `.env` и реальные токены.
- **FR-011**: При отсутствии конфига генерации `make generate` MUST завершаться с ненулевым кодом и понятным сообщением (не silently skip).
- **FR-012**: Документация в `docs/openapi-codegen.md` и `docs/makefile.md` MUST оставаться согласованной с фактическими таргетами после реализации (канон запуска — Make).
- **FR-013**: В scope F03 система MUST NOT требовать автоматизированной проверки «повторный `make generate` → чистый git working tree» (no-diff / drift gate); достаточны успешная генерация и закоммиченные артефакты.

### Key Entities

- **OpenAPI Snapshot**: Локальная копия контракта API (JSON/YAML в `docs/api/`), источник истины для генерации и подсчёта operations.
- **Codegen Config**: Описание параметров генерации клиента (пакет, models/client, путь вывода).
- **Generated API Client**: Автоматически полученный набор типов и HTTP-методов по всем operations снимка; не редактируется вручную как источник поведения.
- **Coverage Expectation**: Зафиксированное ожидаемое число HTTP-операций и матрица соответствия операций целевым CLI-командам (`docs/api/coverage.md`).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: В 100% прогонов на валидном снимке и валидном конфиге `make generate` завершается кодом 0 и оставляет в `internal/apiclient/` сгенерированный клиент, пригодный для последующего адаптера (F04); итоговая поставка F03 включает эти файлы в git.
- **SC-002**: `make api-coverage-check` на каноническом снимке с 51 operation завершается кодом 0; при намеренном несовпадении ожидания — в 100% случаев ненулевой код.
- **SC-003**: Разработчик может обновить снимок и перегенерировать клиент (`openapi-fetch` → `api-coverage-check` → `generate`) без копирования one-liner’ов из чата — только именованные Make-таргеты; каждый таргет при этом вызывается отдельно (нет жёсткой Make-зависимости generate от check).
- **SC-004**: Офлайн (без upstream), имея закоммиченный снимок и конфиг, `make generate` успешно завершается в 100% прогонов на чистом окружении с установленным инструментом генерации.
- **SC-005**: В артефактах этой фичи нет ручных HTTP-обёрток сущностей и нет новых CLI-команд бизнес-CRUD — 0 таких добавлений в scope F03.
- **SC-006**: После успешной генерации сгенерированный клиент отражает полный набор operations текущего снимка (согласованность с проверкой coverage-check на том же снимке).

## Assumptions

- Фича зависит от F01 (каркас репозитория, Makefile, `.env.example`); часть таргетов `openapi-fetch` / `api-coverage-check` / заглушка `generate` уже может существовать — F03 доводит пайплайн до рабочего codegen (конфиг + успешный `generate` + согласованный coverage-check).
- Upstream снимка: публичные URL OpenAPI JSON/YAML SingularityApp API v2; base URL и пути задаются через Makefile/`.env` согласно `docs/openapi-codegen.md`.
- Генератор — `oapi-codegen` (или эквивалент, зафиксированный constitution); установка инструмента — ответственность разработчика/CI (`go install` или аналог), не часть runtime `singctl`.
- Ожидание **51** operations актуально на дату снимка 2026-07-15; при изменении upstream число и матрица обновляются осознанно в том же изменении, что и снимок.
- Auth scheme `Bearer` / адаптеры retry/mapping ошибок — F04; F03 только codegen-клиент.
- Сгенерированный код MAY исключаться из coverage gate (constitution IX); тесты на сам пайплайн (проверка таргетов / счётчик operations) — по необходимости в plan/tasks.
- Коммит `*.gen.go` — обязательная часть DoD F03 (не опциональный артефакт «когда удобно»).
- `generate` и `api-coverage-check` намеренно независимы; рекомендуемый порядок — документация, не Make prerequisite.
- Матрица `docs/api/coverage.md` уже описывает целевые CLI-имена; F03 не реализует эти команды, только поддерживает согласованность снимка с ожидаемым числом operations.
- Глубокая валидация содержимого `coverage.md` (строки/operationId) — вне строгого check F03; при необходимости отдельная доработка позже.
- Автоматический drift-check (повторный generate без diff) отложен за пределы F03; дисциплина «не коммитить рассинхрон снимок/клиент» — на ревью до появления gate.

## Out of Scope

- Ручные HTTP-обёртки, DTO и CRUD-клиенты, написанные не через codegen.
- CLI-команды сущностей (`task`, `project`, `habit`, …) и любые пользовательские команды кроме уже существующих Make-таргетов разработки.
- API adapter, auth header wiring, retry, маппинг ошибок (F04).
- TUI, форматы вывода CLI, CRUD UX.
- Изменение constitution или перенос ожиданий coverage без обновления снимка/матрицы.
- Автоматическая установка `oapi-codegen` в `singctl` binary / встраивание генератора в runtime CLI.
- Автоматический drift/no-diff gate на повторный `make generate` (pre-commit/CI) — отложено после F03.
