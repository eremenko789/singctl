# Feature Specification: Output Rendering

**Feature Branch**: `006-output-rendering`

**Created**: 2026-07-16

**Status**: Draft

**Input**: User description: "F06 Output rendering — форматтеры table|json|yaml|csv; авто-no-color в non-TTY; настройка date_format. Depends F01. Out of scope: конкретные колонки entity (F08+). Acceptance: одни и те же данные рендерятся во все форматы; pipe без ANSI. Inputs: docs/tz §9, constitution."

## Clarifications

### Session 2026-07-16

- Q: Как представлять даты в `json`/`yaml`? → A: Во всех форматах (`table`/`csv`/`json`/`yaml`) даты — строки по `output.date_format`.
- Q: Форма верхнего уровня для `json`/`yaml`? → A: Корень — массив объектов (`[{...}, {...}]` / YAML-список).
- Q: Пустая / отсутствующая дата? → A: В `json`/`yaml` — `null`; в `table`/`csv` — пустая ячейка.
- Q: Уважать переменную окружения `NO_COLOR`? → A: Да — любое непустое значение `NO_COLOR` отключает цвет (как `--no-color`).
- Q: Поверхность проверки DoD F06? → A: Только unit/harness на fixture — без новой CLI-команды.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Same data in all output formats (Priority: P1)

Пользователь (или скрипт) получает один и тот же логический набор строк/полей в любом из форматов `table`, `json`, `yaml`, `csv`. Смена формата меняет только представление, не состав и смысл данных.

**Why this priority**: Acceptance F06 («одни и те же данные рендерятся во все форматы») и ТЗ §9; это контракт Scriptability First (constitution V) до появления entity-команд.

**Independent Test**: На фиксированном наборе строк (fixture / тестовый harness без сетевых entity-колонок) прогнать рендер во все четыре формата и сверить: одинаковые поля, одинаковые значения (с учётом правил дат), одинаковое число записей.

**Acceptance Scenarios**:

1. **Given** логический набор из N строк с известными полями (включая хотя бы одно поле даты), **When** данные рендерятся с `--output table`, `json`, `yaml` и `csv`, **Then** в каждом формате присутствуют те же N записей и те же логические поля/значения (имена полей стабильны между форматами).
2. **Given** тот же набор данных, **When** пользователь (или тест) сравнивает машинночитаемые форматы `json` и `yaml`, **Then** структура эквивалентна: корень — массив объектов (по одному на запись), один и тот же набор полей; порядок ключей внутри объекта MAY отличаться, смысл — нет. Пустой набор → пустой массив `[]` / пустой YAML-список.
3. **Given** тот же набор данных, **When** выбран `csv`, **Then** первая строка — заголовки колонок, соответствующие логическим полям, а последующие строки — значения тех же записей в том же порядке колонок.
4. **Given** пустой набор записей, **When** выполняется рендер во все форматы, **Then** результат валиден для формата (пустая таблица / пустой массив / пустой YAML-список / CSV только с заголовком) и не содержит «фантомных» строк данных.

---

### User Story 2 - Pipe-safe output without ANSI (Priority: P1)

При перенаправлении stdout в файл или pipe цветной ANSI-вывод не попадает в поток данных, чтобы скрипты (`jq`, `awk`, редирект в файл) получали чистый текст.

**Why this priority**: Acceptance F06 («pipe без ANSI») и ТЗ §10 / constitution V; без этого JSON/CSV в pipe ломаются управляющими последовательностями.

**Independent Test**: Рендер с включённым цветом в конфиге при stdout non-TTY (симуляция pipe/redirect); проверить отсутствие ANSI escape-последовательностей в выводе. Отдельно: явный `--no-color` или непустое `NO_COLOR` на TTY также даёт вывод без ANSI.

**Acceptance Scenarios**:

1. **Given** stdout не является интерактивным терминалом (pipe или redirect), **When** выполняется рендер табличного/цветного вывода, **Then** в stdout нет ANSI escape-последовательностей (эквивалент авто-`--no-color`).
2. **Given** stdout — TTY и цвет разрешён конфигурацией, **When** пользователь указывает `--no-color` **или** задаёт непустое `NO_COLOR` в окружении, **Then** цветной вывод отключён (нет ANSI в stdout).
3. **Given** stdout — TTY, цвет в конфиге включён, `--no-color` не задан и `NO_COLOR` пуст/не задан, **When** рендерится `table` (или иной формат, допускающий декорацию), **Then** цвет MAY применяться; для `json`/`yaml`/`csv` цвет MUST NOT искажать синтаксис (предпочтительно без ANSI в машинных форматах).

---

### User Story 3 - Choose format via flag or config (Priority: P2)

Пользователь выбирает формат глобальным `--output`/`-o` или значением по умолчанию из конфигурации `output.format`, не задавая формат на каждой команде.

**Why this priority**: ТЗ §5.1 / §6 (флаг и секция `output`); F01 уже принимает флаг, F02 хранит `output.format` — F06 связывает их с фактическим рендером.

**Independent Test**: При фиксированных данных: (a) явный `-o json` даёт JSON даже если в конфиге `table`; (b) без флага используется `output.format` из effective config; (c) без флага и без значения в конфиге — `table`.

**Acceptance Scenarios**:

1. **Given** в конфиге `output.format: table`, **When** пользователь указывает `-o json`, **Then** stdout в формате JSON.
2. **Given** в конфиге `output.format: csv` и флаг `--output` не передан, **When** выполняется рендер списка, **Then** stdout в формате CSV.
3. **Given** конфиг без `output.format` (или значение по умолчанию) и флаг не передан, **When** выполняется рендер, **Then** используется формат `table`.

---

### User Story 4 - Consistent date display via date_format (Priority: P2)

Даты во всех форматах вывода отображаются согласно настройке `output.date_format` (шаблон в стиле примера ТЗ, по умолчанию день-месяц-год как `2006-01-02`).

**Why this priority**: Явный scope F06 и поле конфига из ТЗ §5.1; без единой политики дат entity-команды (F08+) разойдутся в представлении.

**Independent Test**: Fixture с известным моментом времени; при двух разных `date_format` сравнить отформатированные строки во всех четырёх форматах (`table`/`csv`/`json`/`yaml`).

**Acceptance Scenarios**:

1. **Given** `output.date_format` равен значению по умолчанию из ТЗ (`2006-01-02` как шаблон), **When** рендерится запись с датой, **Then** во всех форматах (`table`/`csv`/`json`/`yaml`) дата — одна и та же строка вида `YYYY-MM-DD` для этой даты.
2. **Given** пользователь задал другой валидный `output.date_format`, **When** рендерится та же запись, **Then** строка даты следует новому шаблону во всех четырёх форматах (включая `json`/`yaml` как строковые значения, не отдельный машинный ISO).
3. **Given** поле даты отсутствует (null/пусто), **When** выполняется рендер, **Then** в `json`/`yaml` значение поля — `null`, а в `table`/`csv` — пустая ячейка; рендерер не падает.

---

### Edge Cases

- Недопустимое значение `--output` уже отклоняется на уровне F01; F06 MUST NOT ослаблять эту валидацию.
- Пустой набор записей: валидный пустой результат по формату, без ошибки.
- Значения с разделителями CSV (запятые, кавычки, переводы строк): CSV MUST корректно экранировать поля, чтобы число логических колонок сохранялось.
- Специальные символы в JSON/YAML (кавычки, unicode): вывод MUST быть синтаксически валидным для формата.
- Невалидный или пустой `output.date_format` в effective config: использовать безопасный default (`2006-01-02`); MUST NOT паниковать.
- Отсутствующая дата: в `json`/`yaml` — JSON/YAML `null`; в `table`/`csv` — пустая ячейка (не литерал `n/a` / `-`).
- Только stdin перенаправлен, а stdout — TTY: авто-отключение цвета по F06 привязано к **stdout** non-TTY (pipe/redirect данных); интерактивный вывод на TTY MAY оставаться цветным, если нет `--no-color` / непустого `NO_COLOR` / `output.color: false`.
- Непустое `NO_COLOR` в окружении MUST отключать цвет независимо от TTY (эквивалент `--no-color`); пустая строка или отсутствие переменной — не влияет. `FORCE_COLOR` вне scope F06.
- Ошибки и диагностические сообщения MUST идти в stderr; полезные данные команды — в stdout (чтобы pipe не смешивал ошибки с данными).
- Конкретные колонки сущностей (`task`, `project`, …) и их подписи — вне scope (F08+); F06 проверяется на обобщённой табличной модели / fixture (unit/harness), без новой пользовательской CLI-команды и без обязательной миграции `config show`.
- Полная матрица exit codes и сценарии scriptability поверх entity-команд — зона F07; F06 фокусируется на форматах и цвете.
- Миграция ad-hoc рендера `config show` (F02) на общий слой — желательна при удобстве, но не обязательна для DoD F06; скрытая/dev demo-команда вывода — вне DoD.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST предоставлять единый механизм рендера логического набора записей (строки + именованные поля) в форматы `table`, `json`, `yaml`, `csv`.
- **FR-001a**: Для `json` и `yaml` корень MUST быть массивом/списком объектов (по одному объекту на запись); MUST NOT использовать объект-обёртку и MUST NOT менять форму в зависимости от числа записей (включая ноль → пустой массив/список).
- **FR-002**: Для одного и того же входного набора все четыре формата MUST отражать одинаковый состав записей и одинаковые логические значения полей (с учётом правил форматирования дат и CSV-экранирования).
- **FR-003**: Формат вывода MUST определяться в порядке приоритета: (1) явно заданный глобальный `--output`/`-o`; (2) иначе `output.format` из effective configuration; (3) иначе `table`.
- **FR-004**: При stdout, который не является TTY, система MUST автоматически отключать цветной ANSI-вывод (поведение, эквивалентное `--no-color` для потока данных).
- **FR-005**: Явный флаг `--no-color` MUST отключать ANSI в stdout независимо от TTY и от `output.color` в конфиге.
- **FR-005a**: Непустое значение переменной окружения `NO_COLOR` MUST отключать ANSI в stdout (та же семантика, что `--no-color`); отсутствие переменной или пустое значение MUST NOT само по себе менять политику цвета. `FORCE_COLOR` вне scope F06.
- **FR-006**: Если цвет разрешён (TTY, нет `--no-color`, нет непустого `NO_COLOR`, `output.color` не запрещает), система MAY использовать цвет в человекочитаемом `table`; машинные форматы `json`, `yaml`, `csv` MUST NOT содержать ANSI escape-последовательностей.
- **FR-007**: Система MUST применять `output.date_format` из effective configuration к полям даты во **всех** форматах вывода (`table`, `csv`, `json`, `yaml`): даты MUST сериализоваться как строки по шаблону (не как отдельный машинный ISO-8601 в `json`/`yaml`); default шаблона — как в ТЗ (`2006-01-02` → отображение `YYYY-MM-DD`).
- **FR-007a**: При отсутствии значения даты система MUST выводить `null` в `json`/`yaml` и пустую ячейку в `table`/`csv` (без текстовых маркеров вроде `n/a` или `-`).
- **FR-008**: При отсутствии/невалидности `date_format` система MUST использовать default шаблон и продолжить рендер без сбоя.
- **FR-009**: DoD F06 MUST закрываться **только** unit/harness-тестами на fixture-данных (доказательство FR-001…FR-008 и связанных FR по цвету/датам); MUST NOT требовать новой пользовательской CLI-команды, скрытой demo-команды или обязательной проводки через `config show`; конкретные колонки entity — F08+.
- **FR-010**: Рендер MUST писать полезные данные в stdout; сообщения об ошибках рендера/валидации — в stderr.
- **FR-011**: CSV-рендер MUST включать строку заголовков и корректно экранировать поля с разделителями/кавычками/переводами строк.
- **FR-012**: Пустой набор записей MUST рендериться во все форматы без ошибки и без выдуманных data-rows.
- **FR-013**: Контракт глобальных флагов `--output`/`-o` и `--no-color` из F01 MUST сохраняться; F06 добавляет семантику рендера, не ломая парсер.
- **FR-014**: Слой рендера MUST быть пригоден для повторного использования будущими CLI list/show командами (и косвенно TUI не обязан использовать те же текстовые форматы, но не должен дублировать политику дат/форматов вразрез с конфигом без причины).

### Key Entities

- **Logical Record Set**: Набор однотипных записей с именованными полями (колонками); источник правды для всех форматов вывода; в `json`/`yaml` сериализуется как корневой массив объектов.
- **Output Format**: Выбранный способ представления (`table` | `json` | `yaml` | `csv`).
- **Color Policy**: Эффективное решение «цвет вкл/выкл» на основе TTY stdout, `--no-color`, непустого `NO_COLOR` и `output.color`.
- **Date Format Setting**: Шаблон отображения дат из `output.date_format` (effective config) с безопасным default.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: На одном fixture-наборе из ≥3 записей с ≥1 полем даты все четыре формата дают согласованные данные (одинаковое число записей и совпадение значений полей после нормализации) в 100% прогонов тестового harness.
- **SC-002**: При симуляции pipe/redirect (stdout non-TTY) доля выводов с ANSI escape в stdout равна 0% для всех четырёх форматов при включённом `output.color` в конфиге.
- **SC-003**: Явный `--no-color` **или** непустое `NO_COLOR` на TTY даёт 0 ANSI-последовательностей в stdout в проверяемых сценариях table-рендера.
- **SC-004**: Смена только `output.date_format` при тех же входных датах изменяет отображаемые строки дат предсказуемо во всех четырёх форматах (два разных валидных шаблона → два разных ожидаемых литерала) без изменения прочих полей.
- **SC-005**: Приоритет формата (флаг > конфиг > `table`) подтверждается тремя автоматическими сценариями без ручной интерпретации.
- **SC-006**: Контракт F06 проверяется одним локальным тестовым прогоном unit/harness на fixture — без entity-команд и без новой CLI-команды вывода.

## Assumptions

- Зависит от F01 (глобальные флаги `--output`/`-o`, `--no-color`) и использует поля `output.*` из модели конфига F02; сетевой API-слой (F04/F05) для DoD не обязателен.
- Конкретные колонки и подписи сущностей SingularityApp появятся в F08+; в F06 достаточно обобщённой табличной модели и тестов.
- Авто-`no-color` определяется по **stdout** (не по stdin): именно stdout уходит в pipe/файл. Дополнительно цвет выключается непустым `NO_COLOR` или флагом `--no-color`.
- В машинных форматах (`json`/`yaml`/`csv`) ANSI не используется даже на TTY.
- Для list/record-set вывода `json`/`yaml` всегда корневой массив объектов (согласовано с pipe-примерами ТЗ вроде `jq '.[].id'`); одиночная «карточка» сущности вне этого контракта F06 (уточнят F08+ при show-командах).
- Default `output.format` = `table`, default `output.color` = true, default `output.date_format` = шаблон `2006-01-02` — согласовано с ТЗ §5.1 и F02.
- Шаблон `date_format` интерпретируется в стиле reference-layout из примера ТЗ (как в документации конфигурации); точная библиотечная привязка — на этапе plan. В `json`/`yaml` даты — те же отформатированные строки, что и в `table`/`csv` (не ISO-only ветка); отсутствующая дата — `null` в `json`/`yaml` и пустая ячейка в `table`/`csv`.
- Ad-hoc форматтеры `config show` (F02) MAY остаться отдельно в F06; унификация — follow-up, не блокер acceptance. DoD = unit/harness only (без demo CLI).
- Стабильные exit codes для скриптов поверх команд и расширенная pipe-матрица — F07; F06 закрывает форматы, цвет и даты.
- Язык заголовков/подписей fixture MAY быть русским или английским; для entity-команд язык UX уточнят поздние фичи, на контракт форматов это не влияет.
