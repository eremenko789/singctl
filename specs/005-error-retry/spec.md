# Feature Specification: Error Handling & Retry

**Feature Branch**: `005-error-retry`

**Created**: 2026-07-16

**Status**: Draft

**Input**: User description: "F05 Error handling & retry — маппинг 401/403/404/422/429/5xx; exponential backoff для 429 (3 попытки); клиентские ошибки (нет токена, неверная дата). Depends F04. Out of scope: TUI-баннеры (F18+). Acceptance: таблица кодов → сообщения; тесты retry на 429. Inputs: docs/tz §8, constitution."

## Clarifications

### Session 2026-07-16

- Q: Политика `Retry-After` при 429? → A: Учитывать `Retry-After` (секунды или HTTP-date), если валиден, с жёстким верхним пределом одной паузы; иначе — чистый exponential backoff.
- Q: Сообщение после исчерпания retry при 429? → A: `Error: rate limited. Retry later`
- Q: Глубина DoD F05 для CLI? → A: Библиотека (taxonomy/retry) + проводка в существующий CLI (`config validate` + общий exit-helper); без новых CRUD-команд.
- Q: DoD для ошибки «неверная дата»? → A: Только общий ParseDate/тип ошибки + unit-тесты; без CLI-флага/команды в F05.
- Q: Код выхода при транспортной ошибке? → A: Exit `1` (как прочие ошибки вызова API).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Predictable HTTP error messages (Priority: P1)

Пользователь CLI (или будущий TUI через тот же слой) при неуспешном ответе API получает стабильное, понятное сообщение по HTTP-коду — без «сырого» `HTTP 401` и без утечки токена.

**Why this priority**: Acceptance F05 требует таблицу кодов → сообщений; это основной UX-контракт ошибок (ТЗ §8.1) и основа scriptability (стабильные тексты/коды выхода).

**Independent Test**: Мок HTTP возвращает по очереди 401, 403, 404, 422, 429 (после исчерпания retry), 5xx; потребитель адаптера получает классифицированную ошибку с ожидаемым user-facing текстом (и при необходимости exit-семантикой) для каждого кода.

**Acceptance Scenarios**:

1. **Given** API возвращает 401, **When** вызов идёт через адаптер с маппингом ошибок, **Then** пользовательское сообщение эквивалентно: `Error: invalid token. Run 'singctl config set-token'` (токен в тексте не раскрывается).
2. **Given** API возвращает 403, **When** ошибка маппится, **Then** сообщение эквивалентно: `Error: insufficient token permissions`.
3. **Given** API возвращает 404 для сущности с известным ID, **When** ошибка маппится, **Then** сообщение эквивалентно: `Error: entity not found: <ID>` (ID — запрошенный идентификатор, если он известен вызывающему контексту; иначе без суффикса ID, но с явной формулировкой «not found»).
4. **Given** API возвращает 422 с телом, содержащим сообщение валидации, **When** ошибка маппится, **Then** пользователю показывается сообщение из тела ответа (не только общий «validation failed»).
5. **Given** API возвращает 5xx, **When** ошибка маппится (без auto-retry для 5xx в F05), **Then** сообщение эквивалентно: `Error: server error, retry later`.

---

### User Story 2 - Automatic retry on rate limit (Priority: P1)

При ответе 429 (rate limit) система автоматически повторяет запрос с exponential backoff до 3 попыток, чтобы кратковременный лимит не ломал одноразовые команды.

**Why this priority**: Явный acceptance F05 («тесты retry на 429») и ТЗ §8.1; без этого CLI непригоден при типичных лимитах API.

**Independent Test**: Мок сначала отвечает 429, затем (на N-й попытке) 2xx или снова 429; тест считает число HTTP-запросов и проверяет, что между попытками есть увеличивающаяся задержка (или контролируемый clock/fake sleep), максимум 3 попытки.

**Acceptance Scenarios**:

1. **Given** мок отвечает 429 на первые две попытки и 2xx на третью, **When** выполняется операция через слой с retry, **Then** операция завершается успехом после ровно трёх HTTP-попыток.
2. **Given** мок отвечает 429 на все запросы, **When** выполняется операция, **Then** выполняется ровно 3 попытки, после чего возвращается классифицированная ошибка rate limit с сообщением эквивалентным `Error: rate limited. Retry later`.
3. **Given** между попытками при 429 без валидного `Retry-After`, **When** измеряется (или инжектируется) backoff, **Then** задержки следуют exponential backoff (каждая следующая пауза строго больше предыдущей в рамках политики F05).
4. **Given** 429 с валидным `Retry-After` в пределах потолка, **When** выполняется retry, **Then** пауза перед следующей попыткой соответствует заголовку (с учётом потолка), а не «слепому» exponential.
5. **Given** мок отвечает 401/404/422/5xx, **When** выполняется операция, **Then** выполняется ровно одна HTTP-попытка (retry только для 429).

---

### User Story 3 - Client-side validation errors (Priority: P2)

До или без сетевого вызова пользователь получает понятные сообщения о локальных ошибках: нет токена и неверный формат даты.

**Why this priority**: ТЗ §8.2 и backlog F05; снижает «шум» API-ошибок и даёт actionable подсказки до запроса.

**Independent Test**: Unit-тест общего разбора даты с невалидной строкой — без сети и без CLI-команды; сверить текст с `Expected: YYYY-MM-DD`. Отдельно: путь без токена (фабрика/validate) — без сети.

**Acceptance Scenarios**:

1. **Given** токен отсутствует или пустой, **When** требуется аутентифицированный вызов / создание сеанса, **Then** возвращается клиентская ошибка конфигурации с подсказкой выполнить `singctl config set-token` (без сетевого запроса; согласовано с fail-fast F04).
2. **Given** вызывается общий разбор даты с невалидной строкой, **When** валидация завершается ошибкой, **Then** сообщение содержит подсказку `Expected: YYYY-MM-DD` (или эквивалент с этим ожидаемым форматом); CLI-команда с датой в F05 не требуется.

---

### User Story 4 - Stable exit semantics for scripts (Priority: P2)

Скрипты и pipe-потребители отличают «не найдено», «ошибка конфигурации» и «прочие ошибки API» по стабильной семантике кодов выхода (ТЗ §10), согласованной с маппингом ошибок.

**Why this priority**: Constitution V (Scriptability First); без стабильных exit codes ошибки бесполезны в автоматизации.

**Independent Test**: Для смоделированных ошибок 404, отсутствия токена и прочего API-сбоя через `config validate` (мок) и/или unit-тесты exit-helper проверить соответствие: not found → 3, config → 2, прочие API → 1.

**Acceptance Scenarios**:

1. **Given** классифицированная ошибка «entity not found» (HTTP 404), **When** CLI завершает команду через общий exit-helper, **Then** код выхода 3.
2. **Given** клиентская ошибка отсутствия токена / конфигурации, **When** CLI завершает команду (в т.ч. `config validate` без токена), **Then** код выхода 2.
3. **Given** прочая ошибка API (401/403/422/429 после исчерпания retry/5xx) или транспортная ошибка (timeout / connection refused) при `config validate` или через exit-helper, **When** CLI завершает команду, **Then** код выхода 1.

---

### Edge Cases

- 429 с заголовком `Retry-After`: если значение валидно (delay-seconds или HTTP-date), пауза MUST равняться этому значению, но MUST NOT превышать зафиксированный верхний предел одной паузы (константа — в plan); при отсутствии/невалидном заголовке — чистый exponential backoff.
- Тело 422 пустое или не JSON: показать безопасный fallback («validation failed» / аналог), не падать и не печатать бинарный мусор.
- ID сущности неизвестен при 404: сообщение без плейсхолдера `<ID>`, но с явной формулировкой not found.
- Сетевая ошибка (connection refused, timeout): не маппится как HTTP-код; отдельная транспортная ошибка, без retry 429-политики; user-facing текст понятен («не удалось связаться с API» / аналог); CLI exit code **1**.
- Токен и Authorization MUST NOT попадать в user-facing сообщения и обычные логи ошибок.
- Повторные попытки при 429 MUST использовать тот же запрос (метод, путь, тело, credentials); MUST NOT менять семантику операции.
- Отмена контекста / таймаут сеанса во время backoff: прервать retry, вернуть ошибку отмены/таймаута, не продолжать попытки после отмены.
- TUI-баннеры, статусбар и inline-формы — вне scope (F18+); слой ошибок MUST отдавать классификацию, пригодную для будущего TUI, но не рисовать UI.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST классифицировать не-2xx ответы адаптера (контракт F04: типизированная ошибка с HTTP status) в стабильную taxonomy как минимум: 401, 403, 404, 422, 429, 5xx (и прочие 4xx/неизвестные — безопасный fallback без утечки секретов).
- **FR-002**: Система MUST предоставлять user-facing сообщения для кодов по таблице ТЗ §8.1 (см. User Story 1); тексты MUST быть стабильны для скриптов и документации.
- **FR-003**: Для HTTP 422 система MUST предпочитать сообщение из тела ответа API, если оно извлекаемо; иначе — безопасный fallback.
- **FR-004**: Для HTTP 404 система MUST включать идентификатор сущности в сообщение, когда ID известен вызывающему контексту.
- **FR-005**: При HTTP 429 система MUST выполнять automatic retry; максимум **3** HTTP-запроса на один логический вызов (acceptance: не больше 3 запросов).
- **FR-005a**: Пауза между попытками при 429 MUST вычисляться так: при валидном `Retry-After` — по заголовку (delay-seconds или HTTP-date), но не выше жёсткого потолка одной паузы; иначе — exponential backoff (базовая пауза с удвоением). Точные константы потолка/базы — в plan; тесты MUST покрывать оба пути (с `Retry-After` и без).
- **FR-006**: Retry/backoff MUST применяться только к HTTP 429; ответы 401/403/404/422/5xx и клиентские ошибки MUST NOT вызывать automatic retry в F05.
- **FR-007**: После исчерпания попыток при 429 система MUST вернуть классифицированную ошибку rate limit с user-facing сообщением эквивалентным `Error: rate limited. Retry later` (не «сырой» статус без taxonomy).
- **FR-008**: Система MUST обрабатывать клиентские ошибки: отсутствующий/пустой токен → подсказка `singctl config set-token`; неверный формат даты → подсказка с `Expected: YYYY-MM-DD`.
- **FR-008a**: Контракт неверной даты в DoD F05 MUST быть закрыт общим ParseDate (или эквивалентом) и unit-тестами; MUST NOT требовать новой CLI-команды или флага с датой в F05.
- **FR-009**: Маппинг ошибок и политика retry MUST жить в общем API-слое (над F04), потребляемом CLI и будущим TUI (constitution IV); MUST NOT дублировать taxonomy в каждом UI-пакете.
- **FR-010**: CLI MUST отображать CLI-тексты ошибок из taxonomy через существующие команды; MUST NOT реализовывать TUI-баннеры/статусбар и MUST NOT добавлять CRUD-команды сущностей в F05.
- **FR-010a**: DoD F05 MUST включать проводку taxonomy/retry в `config validate` (те же user-facing сообщения при не-2xx после маппинга) и общий CLI exit-helper, выбирающий код 0/1/2/3 по классу ошибки.
- **FR-011**: Система MUST согласовать семантику кодов выхода с ТЗ §10: `0` успех, `1` ошибка API (включая транспортные сбои вызова), `2` ошибка конфигурации, `3` not found — для ошибок, классифицируемых этим слоем; проверка — unit-тесты helper + сценарии `config validate` с мок-HTTP.
- **FR-011a**: Транспортные ошибки (connection refused, timeout и аналоги без HTTP status) MUST давать exit code `1`, не `2` и не отдельный код.
- **FR-012**: Система MUST включать unit-тесты с мок-HTTP: (a) таблица код → сообщение/класс для 401/403/404/422/429/5xx; (b) retry на 429 (успех после повторов и исчерпание лимита); (c) отсутствие retry на не-429; (d) клиентские ошибки без сети.
- **FR-013**: Тестовые фикстуры токенов MUST соответствовать constitution VII (фиктивные, низкоэнтропийные).
- **FR-014**: Система MUST NOT писать ручные HTTP CRUD/DTO в обход codegen; retry — тонкая политика поверх существующего адаптера/codegen-клиента (constitution III).

### Key Entities

- **HTTP Error Class**: Класс ошибки по статусу (Unauthorized, Forbidden, NotFound, Validation, RateLimited, Server, Other) с user-facing сообщением и семантикой exit code.
- **Retry Policy**: Правила повтора для RateLimited: максимум 3 HTTP-запроса; пауза = `Retry-After` (с потолком) или exponential backoff; уважение отмены контекста.
- **Client Validation Error**: Локальная ошибка до сети (отсутствие токена, неверная дата) с подсказкой пользователю.
- **Mapped API Failure**: Результат маппинга F04 `HTTP Failure Signal` → классифицированная ошибка F05 (без изменения контракта успешного 2xx).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Для каждого из кодов 401, 403, 404, 422, 429 (после исчерпания retry → `Error: rate limited. Retry later`), 5xx автоматизированный тест в 100% прогонов получает ожидаемый класс/сообщение из таблицы ТЗ §8.1 (с допуском на тело 422 и опциональный ID в 404).
- **SC-002**: При последовательности 429→429→2xx тест фиксирует ровно 3 HTTP-запроса и успешный итог в 100% прогонов.
- **SC-003**: При устойчивом 429 тест фиксирует ровно 3 HTTP-запроса и ошибку с сообщением эквивалентным `Error: rate limited. Retry later` в 100% прогонов.
- **SC-004**: При ответах 401/404/422/5xx тест фиксирует ровно 1 HTTP-запрос (0 retry) в 100% прогонов.
- **SC-005**: Сценарии «нет токена» и «неверная дата» в 100% прогонов завершаются понятной клиентской ошибкой с требуемой подсказкой и без сетевого вызова.
- **SC-006**: В поставке F05 нет TUI-баннеров/статусбара — 0 таких UI-добавлений в scope.
- **SC-007**: Разработчик может прогнать все тесты маппинга и retry офлайн (мок-HTTP) одной канонической командой тестов проекта и получить зелёный результат.
- **SC-008**: `config validate` при смоделированном не-2xx (мок) в 100% прогонов завершается ненулевым кодом по семантике taxonomy (не «сырой» необработанный статус без класса) и без ложного успеха.

## Assumptions

- Зависит от F04: типизированная ошибка не-2xx с программно доступным HTTP status (`errors.As` / аналог); F05 наращивает taxonomy и retry поверх этого контракта.
- Retry только для 429 (как в ТЗ §8.1); 5xx — сообщение «retry later» без automatic retry в F05.
- «3 попытки» = максимум 3 HTTP-запроса на один логический вызов при серии 429.
- Пауза при 429: при валидном `Retry-After` — по заголовку с жёстким потолком; иначе exponential backoff (база порядка 1s с удвоением, напр. 1s затем 2s). Точные константы базы/потолка — в plan/research; лимит «макс. 3 HTTP-запроса» не меняется.
- В тестах backoff MUST быть ускоряемым/подменяемым (fake clock / нулевые задержки), чтобы suite оставался быстрым.
- Клиентская ошибка «нет токена» согласована с fail-fast фабрики сеанса F04; F05 унифицирует сообщение/класс/exit code для потребителей.
- Разбор дат в формате `YYYY-MM-DD` — общий хелпер/ошибка + unit-тесты в F05; команды с `--from`/`--to` подключат хелпер позже (вне обязательной CLI-проводки F05).
- Предупреждение «повторяющиеся задачи не поддерживаются API» (ТЗ §8.2) — вне F05 (honest API boundaries на уровне команд сущностей).
- Exit codes применяются на границе CLI через общий exit-helper; API-слой отдаёт классификацию. DoD F05: проводка в `config validate` + helper (без CRUD-команд). Транспорт без HTTP status → exit `1`.
- TUI использует ту же taxonomy позже (F18+); в F05 достаточно стабильных классов/сообщений без UI-виджетов.
- Интеграционные тесты против реального API не обязательны для DoD F05; мок-HTTP достаточен.

## Out of Scope

- TUI: красные баннеры, статусбар, inline-ошибки форм, индикатор ожидания при 429 (F18+).
- Automatic retry для 5xx и прочих кодов (кроме 429).
- Новые CRUD CLI-команды сущностей и форматы вывода (F06+).
- Изменение codegen/OpenAPI пайплайна (F03) и фабрики сеанса auth/base URL/timeout (кроме потребления и наращивания поверх F04).
- Debug verbose HTTP-лог и redact (F29).
- Политики secure-env (F30).
- Предупреждение о неподдерживаемых recurring tasks (команды сущностей / honest boundaries).
