## User Scenarios & Testing *(mandatory)*

### User Story 1 - Discover the tool via help (Priority: P1)

Новый пользователь (или разработчик) запускает CLI, чтобы понять, что это за инструмент и какие команды доступны. Он ожидает краткое описание продукта на русском, список верхнеуровневых команд и подсказку по глобальным флагам.

**Why this priority**: Без рабочей справки нельзя ни онбордить пользователей, ни развивать дальше CLI — это минимальный полезный артефакт скелета.

**Independent Test**: Запустить `singctl --help` (и при необходимости `singctl help`) без конфигурации и сети; убедиться, что справка выводится и entity-команд в ней нет.

**Acceptance Scenarios**:

1. **Given** установленный бинарник `singctl`, **When** пользователь выполняет `singctl --help`, **Then** отображается описание инструмента на русском, доступные команды верхнего уровня и глобальные флаги с русскими описаниями.
2. **Given** установленный бинарник, **When** пользователь выполняет `singctl` без аргументов, **Then** CLI сообщает понятную ошибку (например, что команда не указана или TUI ещё не реализован), завершается с ненулевым кодом и не запускает сеть/TUI; при неизвестной команде — аналогично ошибка с ненулевым кодом и подсказкой к `--help`.
3. **Given** установленный бинарник, **When** пользователь просматривает список команд в справке, **Then** отсутствуют команды сущностей (`task`, `project`, `habit`, `tag`, `time`) и команда запуска TUI.

---

### User Story 2 - Check installed version (Priority: P1)

Пользователь или скрипт CI проверяет, какая версия CLI установлена, чтобы сверять совместимость и отчёты об ошибках.

**Why this priority**: Версия — обязательный диагностический и релизный примитив; без него нельзя поддержать скрипты и поддержку пользователей.

**Independent Test**: Выполнить `singctl version` и `singctl --version` и убедиться, что печатается один и тот же идентификатор версии; сеть не требуется.

**Acceptance Scenarios**:

1. **Given** установленный бинарник, **When** пользователь выполняет `singctl version` или `singctl --version`, **Then** на stdout выводится одна и та же информация: имя CLI, номер версии и метаданные сборки (как минимум commit и/или date).
2. **Given** установленный бинарник, **When** выполняется `singctl version` или `singctl --version`, **Then** команда завершается успешно и не обращается к удалённым сервисам.

---

### User Story 3 - Accept global flags for later use (Priority: P2)

Пользователь (или тесты) передаёт глобальные флаги, которые будут нужны конфигу, форматированию вывода и отладке в последующих фичах. На этапе скелета флаги должны распознаваться корневым CLI, даже если полная бизнес-логика ещё не подключена.

**Why this priority**: Единый контракт глобальных флагов закрепляет UX ТЗ и не даёт последующим командам вводить несовместимые флаги.

**Independent Test**: Передать каждый глобальный флаг корневой команде/`version`/`--help` и убедиться, что парсер принимает значения и не падает с «unknown flag».

**Acceptance Scenarios**:

1. **Given** установленный бинарник, **When** пользователь указывает `--config /tmp/example.yaml`, `--token TOKEN`, `--output json` (или `-o json`), `--no-color`, `--debug` вместе с `version` или `--help`, **Then** флаги принимаются без ошибки «unknown flag».
2. **Given** установленный бинарник, **When** пользователь указывает недопустимое значение `--output` (не из набора `table`, `json`, `yaml`, `csv`) вместе с любой корневой операцией — включая `--help`, `version` и `--version`, **Then** CLI сообщает об ошибке валидации и завершается с ненулевым кодом **вместо** показа справки/версии (без сетевых вызовов).
3. **Given** установленный бинарник, **When** пользователь указывает только глобальные флаги вместе с разрешённой командой скелета, **Then** поведение остаётся локальным: не открывается TUI и не выполняются HTTP-запросы.

---

### Edge Cases

- Вызов `singctl` без аргументов: понятная ошибка (команда не указана / TUI ещё не реализован), ненулевой exit code; TUI не запускается (до отдельной фичи).
- Неизвестный подкомандный путь: ясное сообщение об ошибке, ненулевой exit code, без паники.
- Повторные/короткие формы флагов (`-o` vs `--output`): обе формы работают одинаково для output.
- Пустой или отсутствующий путь в `--config`: на этом этапе достаточно принять флаг; полный резолвинг конфига — в F02 (не блокирует скелет).
- Недопустимый `--output` имеет приоритет над показом `--help` и версии: сначала ошибка валидации и ненулевой exit.
- Цвет и debug в skeleton: флаги принимаются; полное влияние на HTTP-логи и авто-отключение цвета в non-TTY может появиться в более поздних фичах, но контракт флагов уже зафиксирован.
- Справка не должна обещать entity/TUI-команды, которых ещё нет в scope.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST предоставлять исполняемую CLI-точку входа с именем `singctl`.
- **FR-002**: Система MUST отображать справку по корневой команде через `--help` (и стандартный механизм help), включая описание глобальных флагов.
- **FR-003**: Система MUST предоставлять команду `version` и корневой флаг `--version`, печатающие на stdout одну и ту же информацию: имя CLI (`singctl`), номер версии и метаданные сборки (commit и/или date).
- **FR-004**: Система MUST принимать глобальный флаг `--config` со значением пути к файлу конфигурации.
- **FR-005**: Система MUST принимать глобальный флаг `--token` для переопределения токена из конфигурации (хранение и применение токена — вне scope; флаг должен парситься).
- **FR-006**: Система MUST принимать глобальный флаг `--output` / `-o` со значениями `table`, `json`, `yaml`, `csv` и MUST отклонять любые другие значения с ошибкой валидации на всех корневых операциях, включая `--help`, `version` и `--version`.
- **FR-007**: Система MUST принимать глобальный флаг `--no-color` для отключения цветного вывода.
- **FR-008**: Система MUST принимать глобальный флаг `--debug` для включения подробного диагностического режима.
- **FR-009**: Система MUST отклонять неизвестные глобальные флаги с понятным сообщением об ошибке и ненулевым кодом выхода.
- **FR-010**: Система MUST NOT регистрировать в этом релизе фичи команды сущностей (`task`, `project`, `habit`, `tag`, `time`) и команду запуска TUI.
- **FR-011**: Система MUST NOT выполнять сетевые вызовы при `singctl --help`, `singctl version` и `singctl --version`.
- **FR-012**: Корневая структура команд MUST быть расширяемой так, чтобы последующие фичи могли добавлять подкоманды без ломки глобальных флагов и help/version.
- **FR-013**: При вызове `singctl` без аргументов система MUST завершаться с ненулевым кодом и понятным сообщением об ошибке (отсутствие команды / TUI ещё не реализован), без запуска TUI и без сетевых вызовов.
- **FR-014**: Тексты корневой справки (`--help`), описаний глобальных флагов и сообщений об ошибках парсера/валидации MUST быть на русском языке.

### Key Entities

- **Global Options**: Набор корневых параметров сеанса CLI (`config path`, `token override`, `output format`, `no-color`, `debug`), доступных всем будущим командам.
- **Version Identity**: Идентичность установленной сборки: имя CLI, номер версии и метаданные сборки (commit и/или date), отображаемые через `version` / `--version`.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: В 100% прогонов на чистой машине без сети `singctl --help` завершается успешно и выводит русскоязычную справку с перечислением глобальных флагов за менее чем 2 секунды.
- **SC-002**: В 100% прогонов `singctl version` и `singctl --version` печатают одинаковый непустой вывод, содержащий имя CLI, номер версии и хотя бы одно поле метаданных сборки (commit или date), и завершаются с кодом 0 без сетевого доступа.
- **SC-003**: Каждый из глобальных флагов `--config`, `--token`, `--output`/`-o`, `--no-color`, `--debug` принимается парсером (нет ошибки unknown flag) в ручной или автоматизированной проверке.
- **SC-004**: Справка и дерево команд не содержат entity/TUI-команд из out-of-scope; проверка по списку запрещённых имён проходит без ложных срабатываний.
- **SC-005**: Некорректное значение `--output` приводит к понятной ошибке и ненулевому exit code (и не показывает help/version) у ≥95% тестовых неверных значений из заранее заданного набора на операциях `--help`, `version` и `--version`.
- **SC-006**: В 100% прогонов `singctl` без аргументов завершается с ненулевым кодом, печатает сообщение об ошибке и не открывает интерактивный UI.

## Assumptions

- Имя продукта CLI — `singctl` (sa-cli / SingularityApp CLI), согласно проектным материалам.
- В dev-сборках допустимы placeholder-значения метаданных (`unknown` / `dev`) для commit/date, если полная информация о сборке недоступна; поля всё равно присутствуют в выводе.
- Допустимые значения `--output` зафиксированы ТЗ: `table`, `json`, `yaml`, `csv`; значение по умолчанию для будущих команд — `table` (полная логика форматирования сущностей появится позже).
- Полный резолвинг файла конфигурации, запись токена и валидация API — responsibility F02 / F04; в F01 достаточно объявить и принимать `--config` / `--token`.
- Детальное поведение `--debug` (формат логов, редактирование секретов) и авто-`--no-color` в non-TTY уточняются конституцией и поздними фичами; F01 фиксирует наличие флагов в контракте CLI.
- Реализация следует constitution: один Go-бинарник, CLI на базе принятого в проекте command framework; детали стека — предмет `/speckit-plan`, не этой спецификации.
- Пользовательские строки CLI (help/флаги/ошибки) в F01 — на русском; машинно-ориентированные идентификаторы (имена флагов, коды форматов `table|json|yaml|csv`, имя бинарника) остаются латинскими как в ТЗ.
- Entity-команды и TUI появятся отдельными фичами бэклога и не входят в acceptance F01.
- Пока TUI не реализован, вызов без аргументов НЕ запускает TUI (вопреки целевому ТЗ §4/§7): F01 фиксирует ошибку с ненулевым exit; поведение «без аргументов → TUI» будет введено фичей TUI.

## Out of Scope

- Команды сущностей (`task`, `project`, `habit`, `tag`, `time`) и любые CRUD/list-сценарии.
- TUI-режим и интерактивный UI.
- HTTP/API-вызовы, OpenAPI-клиент, авторизация на сервере.
- Чтение/запись файла конфигурации и безопасное хранение токена (F02 / F29).
- Реальные эффекты форматирования таблиц/JSON для сущностей (достаточно контракта флага `--output`).
