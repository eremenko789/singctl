# Feature Specification: Scriptability & Exit Codes

**Feature Branch**: `007-scriptability-exits`

**Created**: 2026-07-16

**Status**: Draft

**Input**: User description: "F07 Scriptability & exit codes — exit codes 0/1/2/3; stdin/pipe; разделение stdout vs stderr. Depends F05, F06. Out of scope: новые команды. Acceptance: документированные коды выхода; pipe-сценарии из ТЗ §10 проходят на уровне контракта. Inputs: docs/tz §10, constitution V."

## Clarifications

### Session 2026-07-16

- Q: Где публиковать таблицу кодов выхода? → A: И в пользовательской документации `docs/…`, и краткое упоминание в корневом `--help`.
- Q: Код выхода при ошибке использования CLI (неизвестная команда/флаг, неверное значение флага)? → A: Exit `1` (не `2`; `2` только для конфигурации).
- Q: Что с stdout при ошибке? → A: Для DoD-покрытых / нестримящих команд при ошибке stdout MUST быть пустым.
- Q: stderr при успешном завершении? → A: На обычном (не debug) пути DoD stderr MUST быть пустым.
- Q: Минимальная поверхность проверки DoD F07? → A: `config show` + `config validate` (exit-матрица) + misuse-флаг + docs/`--help` (+ reuse F06 fixture для pipe/ANSI).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Stable documented exit codes (Priority: P1)

Автор скрипта или CI-пайплайна опирается на опубликованную таблицу кодов выхода CLI: успех, ошибка вызова API (включая транспорт), ошибка конфигурации, сущность не найдена. Поведение согласовано с уже существующей классификацией ошибок и не меняется от команды к команде.

**Why this priority**: Acceptance F07 («документированные коды выхода») и ТЗ §10; без зафиксированного публичного контракта автоматизация ненадёжна.

**Independent Test**: Сверить опубликованную таблицу кодов (docs + корневой `--help`) с фактическим завершением: `config validate` (успех → 0; API/транспорт → 1; нет токена → 2; not found → 3), misuse-флаг → 1; без новых пользовательских команд.

**Acceptance Scenarios**:

1. **Given** команда завершилась успешно, **When** скрипт проверяет код выхода процесса, **Then** код равен `0`.
2. **Given** классифицированная ошибка вызова API (включая транспортную недоступность) или иная ошибка операции, не относящаяся к конфигурации и not-found, **When** CLI завершает процесс, **Then** код выхода `1`.
3. **Given** ошибка конфигурации (например, отсутствует токен), **When** CLI завершает процесс, **Then** код выхода `2`.
4. **Given** классифицированная ошибка «сущность не найдена», **When** CLI завершает процесс, **Then** код выхода `3`.
5. **Given** ошибка использования CLI (неизвестная команда, неизвестный флаг или неверное значение флага), **When** CLI завершает процесс, **Then** код выхода `1` (не `2` и не `3`), сообщение — в stderr.
6. **Given** пользователь или автоматизация читает таблицу кодов в `docs/…` **или** смотрит корневой `singctl --help`, **When** сравнивает с ТЗ §10 и фактическим поведением существующих команд, **Then** значения `0`/`1`/`2`/`3` и их смысл совпадают (в `--help` — краткое упоминание/сводка; полная таблица — в `docs/`).

### User Story 2 - Clean stdout vs stderr for pipes (Priority: P1)

При успехе полезные данные команды идут только в stdout; диагностические и ошибочные сообщения — только в stderr. Скрипт может перенаправить stdout в файл или следующий инструмент без примеси ошибок и подсказок.

**Why this priority**: Constitution V и ТЗ §10 (редирект/`jq`/`awk`); смешение потоков ломает pipe даже при верных exit codes.

**Independent Test**: `config show` на успехе — данные в stdout, stderr пуст; на ошибках `config validate` / misuse — сообщение в stderr, stdout пуст; без новых команд.

**Acceptance Scenarios**:

1. **Given** команда успешно выводит данные (список/конфиг/результат рендера) на обычном (не debug) пути, **When** stdout и stderr собираются раздельно, **Then** полезные данные присутствуют в stdout, а stderr пуст.
2. **Given** команда завершается с ошибкой (конфиг / API / not found / misuse), **When** потоки собираются раздельно, **Then** user-facing сообщение об ошибке в stderr, код выхода ненулевой по таблице US1, а stdout пуст (нет data-payload и нет текста ошибки в stdout).
3. **Given** пользователь выполняет `singctl … > out.txt` (или эквивалент redirect), **When** команда успешна, **Then** файл содержит только полезный stdout (без ANSI при non-TTY — политика F06).

### User Story 3 - Pipe-ready contract for TZ §10 scenarios (Priority: P1)

Сценарии из ТЗ §10 (редирект JSON в файл; цепочка list → фильтр ID → следующая команда; CSV → подсчёт) зафиксированы как контракт scriptability: форматы вывода, отсутствие ANSI в pipe, разделение потоков и exit codes позволяют эти пайплайны. Полный end-to-end с entity CRUD не требуется в F07 (новые команды вне scope), но контракт и проверка на доступном уровне MUST быть выполнены.

**Why this priority**: Acceptance F07 («pipe-сценарии из ТЗ §10 проходят на уровне контракта»); это мост между F06 (форматы/цвет) / F05 (ошибки/exit) и будущими entity-командами (F08+).

**Independent Test**: Контрактный документ + автоматические проверки на fixture/существующих командах: (a) machine-readable stdout без ANSI при non-TTY; (b) JSON/CSV пригодны для типичных потребителей (`jq`-подобный разбор массива / CSV с заголовком); (c) ненулевой exit не смешивает ошибку с data stdout; (d) каждый пример ТЗ §10 явно отображён на требования контракта с пометкой «реализуется entity-командами F08+» там, где команда ещё отсутствует.

**Acceptance Scenarios**:

1. **Given** сценарий ТЗ «вывод списка в JSON-файл», **When** проверяется контракт, **Then** зафиксировано: `--output json` (или эквивалент), stdout = валидный JSON без ANSI при redirect, exit `0` при успехе.
2. **Given** сценарий ТЗ «list → извлечение id → следующая операция», **When** проверяется контракт, **Then** зафиксировано: JSON-корень — массив объектов с полем идентификатора; stdout пригоден для построчной/фильтровой обработки; ошибки промежуточных шагов дают ненулевой exit и сообщение в stderr.
3. **Given** сценарий ТЗ «CSV → агрегация», **When** проверяется контракт, **Then** зафиксировано: `--output csv` даёт заголовок + строки без ANSI в pipe; поля экранированы так, что число колонок стабильно.
4. **Given** сценарий ТЗ «создание из строк файла через xargs», **When** проверяется контракт, **Then** зафиксировано: аргументы командной строки (не обязательный разбор тела из stdin в F07); каждая итерация независимо возвращает exit по таблице US1; F07 не вводит новую create-команду, но описывает ожидаемое поведение для будущих команд.

### User Story 4 - Predictable behavior when stdin is a pipe (Priority: P2)

Когда CLI запускают в неинтерактивном контексте (stdin и/или stdout — pipe/redirect), поведение остаётся предсказуемым для скриптов: нет интерактивных запросов, мешающих автоматизации; политика цвета для потока данных согласована с F06; процесс завершается кодом из таблицы US1.

**Why this priority**: ТЗ §10 требует поддержки pipe; без явных правил non-interactive запуска скрипты зависают на prompt’ах или получают «грязный» вывод.

**Independent Test**: Запуск существующих команд с перенаправленным stdin и/или stdout (или симуляция non-TTY): команда завершается без ожидания интерактивного ввода; цвет в data-stdout отключён при stdout non-TTY (F06); exit codes по US1.

**Acceptance Scenarios**:

1. **Given** stdout перенаправлен в pipe/файл, **When** выполняется существующая команда с данными, **Then** в data-stdout нет ANSI (F06), полезные данные пригодны для дальнейшей обработки.
2. **Given** stdin перенаправлен из файла/pipe, а команда не читает payload из stdin, **When** выполняется команда, **Then** она не блокируется ожиданием ввода и завершается обычным exit-кодом.
3. **Given** non-interactive запуск, **When** возникает ошибка, **Then** CLI не запрашивает подтверждение/ввод для продолжения — сразу stderr + ненулевой exit.

### Edge Cases

- Успех с пустым набором данных (пустой список): exit `0`, валидный пустой результат формата в stdout (F06), без ошибки в stderr.
- Одновременный redirect stdout и stderr: скрипт MAY разделять потоки (`2>`); контракт F07 не смешивает их в один канал по умолчанию.
- Ошибка после частичного вывода: для команд без стриминга (DoD F07) при ненулевом exit stdout MUST быть пустым — не оставлять частичный machine-readable payload; диагностика только в stderr. Политика для будущих стримящих команд — вне F07.
- Неизвестный/необработанный класс ошибки: MUST маппиться в exit `1`, не в `0`.
- Client validation (неверная дата и т.п.): по F05 → exit `1` (не config `2`, не not-found `3`), если это не ошибка конфигурации.
- Ошибка использования CLI (неизвестная команда, неизвестный флаг, неверное значение глобального/локального флага и аналогичный misuse): exit `1`; MUST NOT использовать `2` (зарезервирован для конфигурации) и MUST NOT `3`.
- F07 MUST NOT вводить новые пользовательские CLI-команды; минимальный DoD: `config show` (streams), `config validate` (exit-матрица), один misuse-флаг, docs + корневой `--help`, плюс reuse F06 fixture для pipe/ANSI; полный live E2E `task`/`time` — F08+.
- Полный live E2E примеров ТЗ с `task`/`time` CRUD — зона F08+; в F07 — контрактное покрытие.
- Shell completion, алиасы команд (ТЗ §11) — вне scope F07.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST сохранять и публиковать стабильную семантику кодов выхода: `0` — успех; `1` — ошибка API / операция / транспорт / ошибка использования CLI (неизвестная команда/флаг, неверное значение флага) / прочие не-config и не-not-found ошибки; `2` — ошибка конфигурации; `3` — not found — в соответствии с ТЗ §10 и классификацией F05.
- **FR-001a**: Ошибки использования CLI MUST давать exit `1`; MUST NOT маппиться на `2` или `3`.
- **FR-002**: Документация scriptability MUST включать полную таблицу кодов выхода с кратким смыслом каждого кода в пользовательских `docs/` (без необходимости читать исходники); корневой `singctl --help` MUST кратко упоминать коды `0`/`1`/`2`/`3` (или ссылаться на docs) так, чтобы контракт был обнаруживаем из бинаря.
- **FR-003**: При успехе на обычном (не debug) пути полезные данные команды MUST писаться в stdout; stderr MUST быть пустым; сообщения об ошибках MUST писаться в stderr (при ненулевом exit). Verbose/debug-диагностика на stderr — вне обязательного DoD F07.
- **FR-004**: При ошибке процесс MUST завершаться ненулевым кодом по FR-001; stderr MUST содержать user-facing сообщение; для DoD-покрытых нестримящих команд stdout MUST быть пустым (MUST NOT содержать data-payload и MUST NOT быть единственным каналом текста ошибки).
- **FR-005**: Контракт pipe-сценариев ТЗ §10 MUST быть зафиксирован явно (какие свойства stdout/stderr/exit/формата требуются для каждого примера), с пометкой зависимости от entity-команд там, где команды ещё отсутствуют.
- **FR-006**: DoD F07 MUST доказывать контракт на минимальной поверхности: (1) `config show` — успех: data в stdout, stderr пуст; (2) `config validate` — матрица exit `0`/`1`/`2`/`3`; (3) как минимум один misuse (неизвестный или неверный флаг) → exit `1`, stdout пуст, ошибка в stderr; (4) полная таблица в `docs/` + краткое упоминание в корневом `--help`; (5) reuse F06 fixture/harness для pipe-safe/ANSI. MUST NOT требовать новых пользовательских команд и live entity CRUD E2E.
- **FR-007**: В non-interactive / pipe контексте CLI MUST NOT блокироваться ожиданием интерактивного ввода для обычных ошибок и успешных путей существующих команд.
- **FR-008**: Политика авто-отключения цвета при stdout non-TTY и `--no-color` / `NO_COLOR` MUST оставаться согласованной с F06; F07 не ослабляет эти правила.
- **FR-009**: Машинночитаемые форматы в pipe (`json`/`yaml`/`csv`) MUST оставаться синтаксически пригодными для типичных потребителей пайплайнов (валидный JSON-массив записей / CSV с заголовком), как задано F06.
- **FR-010**: Общий механизм выбора exit-кода (F05) MUST оставаться единственным источником истины для границы процесса; F07 не вводит конкурирующую таблицу кодов.
- **FR-011**: Сценарий «строки файла → многократный вызов команды с аргументами» (xargs-стиль из ТЗ) MUST быть отражён в контракте: независимый exit на каждый вызов; разбор тела задачи из stdin как входного формата create — вне обязательного DoD F07 (нет новой create-команды).

### Key Entities

- **Exit Code Contract**: Публичная таблица `0|1|2|3` и правила отнесения исходов команд к кодам.
- **Stream Separation Policy**: Правила размещения полезных данных (stdout) и диагностики/ошибок (stderr).
- **Pipe Scenario Contract**: Формализация примеров ТЗ §10 как набора проверяемых свойств (формат, ANSI, потоки, exit) до появления entity-команд.
- **Non-Interactive Context**: Запуск с non-TTY stdin и/или stdout (pipe/redirect), без интерактивных prompt’ов.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Полная таблица кодов `0`/`1`/`2`/`3` с описанием смысла опубликована в `docs/` и совпадает с ТЗ §10 (100% кодов); корневой `--help` содержит краткое упоминание тех же кодов (проверяемо по тексту help).
- **SC-002**: Автоматические проверки показывают: успех → exit `0`; config → `2`; not found → `3`; API/транспорт/ошибка использования CLI/прочее → `1` в 100% прогонов harness на существующих путях (без новых команд); как минимум один сценарий misuse (неизвестный флаг или неверное значение флага) → `1`.
- **SC-003**: В успешных обычных (не debug) сценариях с данными stderr пуст в 100% проверяемых тестов, а полезный payload только в stdout; при ошибке user-facing текст в stderr, stdout пуст в 100% проверяемых ошибочных сценариев DoD.
- **SC-004**: Все четыре примера пайплайнов из ТЗ §10 имеют явную запись в контракте F07 со статусом «проверяемо сейчас» или «контракт для F08+» — ни один пример не оставлен без покрытия на уровне контракта.
- **SC-005**: При симуляции stdout non-TTY machine-readable вывод проверяемых сценариев содержит 0 ANSI escape-последовательностей (согласовано с F06).
- **SC-006**: DoD F07 закрывается одним локальным прогоном тестов на поверхности FR-006 (`config show` / `config validate` / misuse / docs+`--help` / F06 fixture) — без новой пользовательской CLI-команды и без обязательного live entity E2E.

## Assumptions

- Зависит от F05 (taxonomy ошибок + `ExitCode`) и F06 (форматы, pipe-safe цвет, stdout для данных); F07 закрепляет scriptability как продуктовый контракт и закрывает пробелы документации/потоков/pipe-сценариев.
- Числовая семантика exit уже задана F05/ТЗ §10; F07 не пересматривает значения кодов, а делает их документированным и единообразно соблюдаемым контрактом.
- Новые entity-команды (`task`, `time`, …) вне scope; примеры ТЗ §10 с ними закрываются контрактом сейчас и реализацией в F08+.
- «Поддержка stdin/pipe» в F07 означает предсказуемый non-interactive запуск и пригодность stdout для пайплайнов, а не обязательный JSON-stdin API для create в этой фиче.
- Интерактивные confirm/prompt для разрушающих операций, если появятся позже, MUST иметь non-interactive обход (флаг/отказ) — детализация в фичах команд; для текущих команд F07 требует отсутствия блокирующих prompt’ов.
- Язык документации кодов выхода MAY быть русским или английским; важна однозначность смысла кодов.
- Минимальная поверхность DoD: `config show`, `config validate`, один misuse-флаг, docs + корневой `--help`, reuse F06 pipe/ANSI fixture; остальные существующие подкоманды MAY быть покрыты дополнительно, но не обязательны для acceptance F07.
- Shell completion и алиасы (ТЗ §11) не входят в F07.
