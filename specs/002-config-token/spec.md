# Feature Specification: Config & Token Storage

**Feature Branch**: `002-config-token`

**Created**: 2026-07-16

**Status**: Draft

**Input**: User description: "F02 Config & token storage — команды config set-token|show|validate|set; резолвинг конфига (флаг → ./.singctl.yaml → XDG → ~/.config); формат config.yaml. Depends F01. Out of scope: CRUD сущностей; углублённое маскирование токена (F29)."

## Clarifications

### Session 2026-07-16

- Q: Какой формат токена принимает `set-token` и что хранится в `api.token`? → A: `set-token` принимает только «голый» токен без префикса `Bearer`; в файл пишется значение без `Bearer`; префикс `Bearer` добавляется при формировании HTTP-заголовка `Authorization`.
- Q: Учитывает ли `config show` глобальный флаг `--output`? → A: Да; `config show` уважает `--output` (`table|json|yaml|csv`); по умолчанию (без флага) — человекочитаемый YAML.
- Q: Какой формат маскирования токена в `config show`? → A: В `config show` токен маскируется форматом `первые 4 символа + **** + последние 4 символа` (если токен слишком короткий — показывается только `****`).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Save API token locally (Priority: P1)

Оператор SingularityApp сохраняет API-токен через CLI, чтобы последующие команды могли аутентифицироваться без повторного ввода секрета в каждой команде.

**Why this priority**: Без локального хранения токена остальные authenticated-команды неработоспособны; это базовый онбординг после установки CLI.

**Independent Test**: Выполнить `singctl config set-token <TOKEN>` на машине без сети; убедиться, что токен записан в выбранный файл конфигурации и может быть прочитан обратно (через `config show` с маскированием или через изолированную проверку чтения).

**Acceptance Scenarios**:

1. **Given** установленный CLI без файла конфигурации, **When** пользователь выполняет `singctl config set-token <TOKEN>`, **Then** создаётся (при необходимости) каталог и файл конфигурации в каноническом месте записи по умолчанию, токен сохранён в секции API, команда завершается успешно.
2. **Given** уже существующий файл конфигурации, найденный по правилам резолвинга, **When** пользователь выполняет `singctl config set-token <NEW_TOKEN>`, **Then** токен в файле обновляется, остальные поля конфигурации сохраняются.
3. **Given** пользователь указал `--config /path/to/custom.yaml`, **When** выполняется `singctl config set-token <TOKEN>`, **Then** токен записывается именно в указанный файл (файл создаётся при отсутствии).

### User Story 2 - Inspect effective configuration (Priority: P1)

Пользователь проверяет, какая конфигурация сейчас действует (base URL, формат вывода, наличие токена), не раскрывая секрет полностью.

**Why this priority**: Диагностика «почему не работает API» начинается с просмотра конфига; без safe `show` онбординг и поддержка невозможны.

**Independent Test**: Подготовить файл с известными значениями и токеном; выполнить `singctl config show` и убедиться, что поля видны, а значение токена замаскировано (не печатается целиком).

**Acceptance Scenarios**:

1. **Given** валидный файл конфигурации с токеном и base URL, **When** пользователь выполняет `singctl config show`, **Then** выводятся действующие значения конфигурации (как минимум base URL, timeout, настройки output), а токен отображается в замаскированном виде.
2. **Given** токен передан глобальным флагом `--token`, **When** пользователь выполняет `singctl config show`, **Then** отображается, что override токена активен, при этом полный секрет не печатается.
3. **Given** конфигурация не найдена ни по одному пути резолвинга и override токена не задан, **When** выполняется `singctl config show`, **Then** CLI сообщает, что конфиг отсутствует / пуст, завершается предсказуемо (без паники) и не раскрывает секретов.
4. **Given** валидная конфигурация, **When** пользователь выполняет `singctl config show -o json`, **Then** вывод — валидный JSON с действующими значениями и замаскированным токеном; без `-o` — человекочитаемый YAML.

### User Story 3 - Resolve configuration from multiple locations (Priority: P1)

Пользователь (или CI) задаёт конфиг через `--config`, XDG, `~/.config` или локальный `.singctl.yaml` и ожидает единый приоритет: более специфичный источник побеждает.

**Why this priority**: Предсказуемый резолвинг — контракт ТЗ §5 и основа для скриптов/CI; без него команды читают «не тот» файл.

**Independent Test**: Разместить разные файлы на уровнях приоритета и проверить, какой файл реально используется командами `config show` / `config set` при наличии/отсутствии `--config`.

**Acceptance Scenarios**:

1. **Given** указан `--config /path/to/file.yaml`, **When** выполняется любая команда `config *`, **Then** используется только указанный путь (остальные кандидаты игнорируются).
2. **Given** `--config` не указан, но существует `./.singctl.yaml` в cwd, **When** выполняется `config show`, **Then** читается этот файл, даже если ниже по приоритету есть XDG и `~/.config` варианты.
3. **Given** cwd-файла нет, но существует `$XDG_CONFIG_HOME/singctl/config.yaml`, **When** выполняется `config show`, **Then** читается этот файл, даже если ниже по приоритету есть `~/.config/singctl/config.yaml`.
4. **Given** отсутствуют `--config`, cwd и XDG варианты, но есть `~/.config/singctl/config.yaml`, **When** выполняется `config show`, **Then** используется `~/.config/singctl/config.yaml`.

---

### User Story 4 - Set arbitrary config keys (Priority: P2)

Пользователь меняет отдельные параметры (например, base URL, timeout, формат вывода) без ручного редактирования YAML.

**Why this priority**: `set-token` покрывает главный секрет; `set` нужен для повседневной настройки, но онбординг возможен и без него через ручной файл.

**Independent Test**: Выполнить `singctl config set <key> <value>`, затем `config show` и убедиться, что значение отражено в действующей конфигурации.

**Acceptance Scenarios**:

1. **Given** существующий или отсутствующий файл конфигурации, **When** пользователь выполняет `singctl config set api.base_url https://example.invalid`, **Then** значение сохраняется и видно в `config show`.
2. **Given** неизвестный или недопустимый ключ/значение, **When** выполняется `singctl config set ...`, **Then** CLI сообщает понятную ошибку и не портит существующий файл (запись не применяется).
3. **Given** указан `--config`, **When** выполняется `config set`, **Then** изменение пишется в указанный файл.

---

### User Story 5 - Validate API connectivity (Priority: P2)

Пользователь проверяет, что сохранённый (или override) токен позволяет подключиться к API, либо получает ясный статус, если проверка ещё на заглушке до появления HTTP-клиента.

**Why this priority**: Validate — быстрый smoke после `set-token`; полный клиент может появиться позже (F04), поэтому допустима заглушка с честным сообщением.

**Independent Test**: При наличии заглушки — убедиться, что `config validate` завершается предсказуемо и сообщает статус; при наличии API-слоя — успешный/неуспешный токен даёт различимый результат.

**Acceptance Scenarios**:

1. **Given** токен задан в конфиге или через `--token`, и доступен работающий слой проверки API, **When** пользователь выполняет `singctl config validate`, **Then** при валидном токене команда сообщает об успехе (код 0), при невалидном — понятную ошибку и ненулевой код.
2. **Given** токен отсутствует, **When** выполняется `singctl config validate`, **Then** CLI сообщает, что токен не задан, подсказывает `config set-token`, завершается с ненулевым кодом.
3. **Given** полноценный API-клиент ещё недоступен, **When** выполняется `singctl config validate`, **Then** допускается заглушка: команда либо выполняет локальную проверку наличия токена/формата конфига, либо явно сообщает, что удалённая проверка отложена — без CRUD-вызовов сущностей и без ложного «OK» при отсутствии реальной проверки.

---

### Edge Cases

- Пустой токен / только пробелы в `set-token`: отказ с понятной ошибкой, файл не создаётся/не затирается пустым секретом.
- Токен с префиксом `Bearer` (с пробелом после) в `set-token`: отказ с понятной ошибкой (команда принимает только «голый» токен); пользователю подсказывается передать значение без префикса.
- Файл конфигурации существует, но содержит некорректный YAML или отсутствующие обязательные секции: понятная ошибка чтения без паники.
- Нет прав на запись в целевой каталог/файл: ошибка I/O с ненулевым кодом.
- Одновременно заданы `--token` и токен в файле: для runtime-операций (включая `validate`) приоритет у `--token`; `set-token` всё равно пишет в файл.
- `XDG_CONFIG_HOME` пуст или не задан: используется fallback `~/.config/singctl/config.yaml` согласно порядку ТЗ.
- Локальный `./.singctl.yaml` в cwd имеет приоритет над XDG и `~/.config` при чтении.
- `config show` печатает токен только в замаскированном виде (первые 4 + `****` + последние 4; для слишком коротких токенов — только `****`), и никогда не выводит полный секрет; углублённый redact в debug-логах — F29.
- Параллельная запись в один файл не гарантируется (один пользователь/процесс — нормальный кейс); при ошибке записи состояние не должно становиться частично обрезанным файлом без сообщения об ошибке.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST предоставлять группу команд `singctl config` с подкомандами `set-token`, `show`, `validate` и `set`.
- **FR-002**: Система MUST резолвить файл конфигурации в порядке приоритета (убывание): (1) флаг `--config`, (2) `./.singctl.yaml`, (3) `$XDG_CONFIG_HOME/singctl/config.yaml`, (4) `~/.config/singctl/config.yaml`.
- **FR-003**: Система MUST поддерживать формат конфигурации с секциями `api` (как минимум `base_url`, `token`, `timeout`), `output` (как минимум `format`, `color`, `date_format`) и `tui` (как минимум `theme`, `vi_keys`, `refresh_interval`) согласно ТЗ §5.1; неизвестные ключи при чтении MAY игнорироваться с сохранением round-trip где это практично.
- **FR-004**: `config set-token <TOKEN>` MUST принимать только «голый» токен без префикса `Bearer` и MUST сохранять его в поле `api.token` выбранного (или создаваемого) файла конфигурации **без** префикса `Bearer`; при передаче значения, начинающегося с `Bearer` (с пробелом после), команда MUST отклонять ввод с понятной ошибкой.
- **FR-004a**: При формировании HTTP-заголовка `Authorization` система MUST добавлять префикс `Bearer` (с пробелом после) к значению из `api.token` (или runtime-override `--token`), если префикса ещё нет.
- **FR-005**: При отсутствии файла конфигурации и отсутствии `--config` команды записи (`set-token`, `set`) MUST создавать файл в каноническом месте по умолчанию: `$XDG_CONFIG_HOME/singctl/config.yaml`, если `XDG_CONFIG_HOME` задан, иначе `~/.config/singctl/config.yaml` (не в `./.singctl.yaml` без явного желания пользователя).
- **FR-006**: `config show` MUST выводить действующую (effective) конфигурацию после резолвинга и применения runtime-override `--token`.
- **FR-006a**: `config show` MUST уважать глобальный флаг `--output` / `-o` со значениями `table`, `json`, `yaml`, `csv`; при отсутствии флага MUST выводить человекочитаемый YAML; токен MUST оставаться замаскированным во всех форматах вывода.
- **FR-007**: `config show` MUST маскировать значение токена так, чтобы вывод никогда не содержал полный секрет: формат `первые 4 символа + **** + последние 4 символа` (если токен короче 8 символов — только `****`). Углублённое маскирование в debug-логах и политиках безопасности — вне scope (F29), но базовое скрытие в `show` обязательно.
- **FR-008**: `config set <key> <value>` MUST устанавливать допустимые ключи конфигурации (dotted path, согласованный со схемой §5.1) и сохранять их в файл по тем же правилам записи, что и `set-token`.
- **FR-009**: `config set` MUST отклонять недопустимые ключи или значения с понятной ошибкой и ненулевым кодом без повреждения существующей конфигурации.
- **FR-010**: `config validate` MUST проверять готовность к работе с API: наличие токена (файл или `--token`) и, при доступности API-слоя, успешность подключения; до появления слоя допускается заглушка с честным статусом (не выдавать успех удалённой проверки, если она не выполнялась).
- **FR-011**: При отсутствии токена `config validate` MUST завершаться с ненулевым кодом и подсказкой использовать `config set-token`.
- **FR-012**: Глобальный флаг `--token` MUST иметь приоритет над токеном из файла для runtime-чтения (effective config), не заменяя сам файл до явного `set-token`.
- **FR-013**: Система MUST NOT выполнять CRUD-операции над сущностями SingularityApp в рамках этой фичи.
- **FR-014**: Тексты help, ошибок и пользовательских сообщений команд `config` MUST быть на русском языке (как в F01); имена команд/флагов/ключей конфига — в каноническом латинском виде из ТЗ.
- **FR-015**: Токен MUST NOT попадать в репозиторий проекта: команды работают с локальными путями пользователя; пример конфигурации в репозитории, если есть, не содержит реальных секретов.

### Key Entities

- **Config Document**: Локальный документ настроек CLI/TUI с секциями `api`, `output`, `tui` и их полями.
- **Resolved Config Path**: Итоговый путь к файлу после применения приоритета `--config` → cwd `.singctl.yaml` → XDG → `~/.config`.
- **Effective Settings**: Снимок настроек после чтения файла и применения runtime-override (`--token` и при необходимости прочих глобальных флагов сеанса).
- **API Token**: Секрет аутентификации к API, хранимый в `api.token` **без** префикса `Bearer` и отображаемый только в замаскированном виде в `config show`; префикс `Bearer` применяется только при HTTP-запросах.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: В 100% прогонов `config set-token` с валидным токеном завершается кодом 0 и приводит к тому, что последующий `config show` отражает наличие замаскированного токена (файл создан или обновлён).
- **SC-002**: Во всех четырёх сценариях приоритета резолвинга (флаг / cwd / XDG / `~/.config`) `config show` читает именно ожидаемый файл — 100% совпадение с матрицей приоритета.
- **SC-003**: `config show` никогда не печатает полный токен из тестового набора секретов и всегда выводит токен в замаскированном формате (`первые 4 + **** + последние 4` или `****` для коротких) (0 утечек полного значения на stdout/stderr в автоматизированной проверке).
- **SC-004**: Пользователь может сохранить токен и убедиться в результате (`set-token` → `show`) менее чем за 30 секунд без ручного редактирования YAML.
- **SC-005**: `config set` для заранее заданного набора допустимых ключей успешно обновляет значения (код 0); для набора недопустимых ключей/значений — 100% отказ с ненулевым кодом без порчи файла.
- **SC-006**: `config validate` при отсутствии токена в 100% случаев даёт ненулевой код и сообщение с подсказкой `set-token`; при наличии токена — либо реальный результат проверки API, либо честный результат заглушки без ложного «удалённо OK».
- **SC-007**: Справка `singctl config --help` и help подкоманд доступны офлайн и завершаются успешно за менее чем 2 секунды.
- **SC-008**: `config show -o json` и `config show -o yaml` в 100% прогонов выдают валидный структурированный вывод с замаскированным токеном; без `-o` — человекочитаемый YAML.

## Assumptions

- Фича зависит от F01 (корневой CLI, глобальные флаги `--config` / `--token`, русскоязычные тексты справки).
- Канонический default path для новой записи: XDG при наличии переменной, иначе `~/.config/singctl/config.yaml` — не cwd-файл, чтобы случайный `set-token` в чужой директории не оставлял секрет в проекте.
- Значение по умолчанию `api.base_url` при создании нового файла: `https://api.singularity-app.ru` (как в ТЗ §5.1); точное значение prod URL может быть уточнено позже по OpenAPI без ломки команд.
- Значение по умолчанию `api.timeout`: 30s; `output.format`: `table`; `output.color`: true; `tui.theme`: `dark`; `tui.vi_keys`: true; `tui.refresh_interval`: 0 — как в примере ТЗ.
- Базовое маскирование токена в `config show` входит в F02 (constitution VII); углублённый redact в `--debug`/логах и доп. политики — F29.
- `config validate` может использовать заглушку до F04 (HTTP/API-слой); после появления клиента — реальная проверка подключения без CRUD сущностей.
- Хранение токена только в локальном конфиге пользователя; интеграция с OS keychain / env-only режимы — вне scope (могут появиться в F29/F30).
- Команды `config` не требуют TUI и не запускают интерактивный wizard в этой фиче (wizard упомянут в ТЗ как целевой UX — отдельная доработка при необходимости).
- Имя файла при `--config` может быть любым; имя по умолчанию в XDG/`~/.config` — `config.yaml`; локальный кандидат — именно `./.singctl.yaml`.
- Пример `token: "Bearer xxxx..."` в ТЗ §5.1 — иллюстративный; фактическое хранение — «голый» токен без префикса `Bearer` (см. Clarifications 2026-07-16).

## Out of Scope

- CRUD/list команды сущностей (`task`, `project`, `habit`, `tag`, `time`) и любые бизнес-операции API кроме минимальной проверки в `validate`.
- Углублённое маскирование/redact токена в debug-логах и расширенные политики credential security (F29).
- Secure env / запрет записи секретов в определённые пути (F30).
- TUI и интерактивный wizard настройки.
- OpenAPI codegen и полноценный HTTP-клиент (F03/F04) — кроме опционального использования для `validate`, когда они уже есть.
- Изменение глобальных флагов скелета F01 (кроме использования уже объявленных `--config` / `--token`).
