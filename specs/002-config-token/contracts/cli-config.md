# CLI Contract: singctl config (F02)

**Feature**: `002-config-token` | **Date**: 2026-07-16

Контракт описывает поведение root CLI `singctl` при добавлении команды группы `config`.
Аудитория: пользователи CLI и автотесты, включая будущие задачи F02+.

---
## Binary

См. контракт root CLI: `specs/001-cli-skeleton/contracts/cli.md`.

---
## Invocation grammar (extension)

```text
singctl config <subcommand> [args...]
```

<subcommand> ::= `set-token` | `show` | `validate` | `set`

Поведение root-level флагов ( `--config`, `--token`, `--output`/`-o`, `--no-color`, `--debug` ) сохраняется как в F01.

---
## Global flags interaction

Для всех `config`-команд:
- `--config` задаёт путь до YAML (приоритет выше остальных источников).
- `--token` влияет на runtime-использование токена (например, при `show`/`validate`), но **не** заменяет файл: изменение файла — только через `set-token`.
- `--output`/`-o` влияет на формат вывода `config show`.

---
## Commands

### `singctl config set-token <TOKEN>`

**Purpose**: сохранить токен в `api.token` выбранного/создаваемого файла конфигурации.

**TOKEN format**:
- принимается только "голый" токен без префикса `Bearer`;
- если пользователь передаёт значение, начинающееся с `Bearer` (с пробелом после), команда отклоняет ввод и завершается ненулевым кодом.

**Writes**:
- создаёт целевой каталог и файл, если файла нет;
- записывает токен в секцию `api.token` без префикса `Bearer`.

**Exit codes**:
- успех: 0
- ошибки: ненулевой код (ошибка валидации ввода или I/O при записи).

---
### `singctl config show`

**Purpose**: показать effective конфигурацию (включая наличие токена) без раскрытия полного секрета.

**Token masking**:
- `первые 4 символа + **** + последние 4 символа` (если длина < 8 — только `****`).

**Output formats**:
- по умолчанию — человекочитаемый формат (YAML);
- при `-o/--output json|yaml|csv|table` — структурированный вывод в выбранном формате.

**Exit codes**:
- успех: 0, если конфигурация резолвится и доступна для чтения;
- ошибка: ненулевой код (например, конфиг отсутствует/пуст и runtime-override не задан) — без паники и без утечек токена.

---
### `singctl config set <key> <value>`

**Purpose**: обновить один параметр конфигурации по dotted path.

**Key format**:
- `<key>` задаётся в форме dotted path, согласованной со схемой `ConfigDocument`.

**Validation**:
- неизвестные ключи или несовместимые значения отклоняются;
- при ошибке запись не должна приводить к повреждению существующей конфигурации.

**Exit codes**:
- успех: 0
- ошибки: ненулевой код.

---
### `singctl config validate`

**Purpose**: проверить готовность CLI к работе с API на основе токена и конфигурации.

**Stub behavior (до появления полного HTTP/API слоя)**:
- если токена нет: ошибка с подсказкой использовать `config set-token`, exit != 0
- если токен есть: локальная проверка считается пройденной, удалённая проверка отложена (команда не должна выдавать ложное "удалённо OK").

**Exit codes**:
- отсутствует токен: ненулевой код
- токен присутствует: 0 (локальная проверка пройдена), удалённая проверка позже
